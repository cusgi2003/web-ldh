SQL * PLUS : 프로그램
ㄴ 오라클에서 제공해주는 명령어 기반 프로그램
ㄴ 오라클 데이터베이스 접속해서 SQL문을 작성하고 실행할 수 있다

SQL 버퍼(Buffer)의 사용
ㄴ 임시 저장 공간이 존재한다
ㄴ 이전에 실행했던 SQL이 저장된다

데이터 조작 프로시저 테스트

디버깅을 잘 해야 한다
디버깅 : 오류의 원인을 찾아 수정하는 과정

프로시저를 실행했는데 예상한 결과와 다르다 -> 그 이유를 찾아야 한다

1. 입력값이 제대로 들어갔는지
2. select결과는 존재하는지
3. update 조건은 맞게 썼는지
4. commiit이 필요한지
5. 예외가 발생하지는 않았는지

SQL * PLUS 명령어 유형


파일
SAVE - 버퍼에 들어있는 내용을 저장
GET - 파일의 내용을 SQL에 저장
SPOOL - 실행 결과를 파일로 저장한다

편집
L - 현재 SQL 버퍼 내용을 확읺나다
A(append) - 현재 줄 끝에 내용을 추가한다
C(change) - 문자열을 변경한다
I(insert) - 새로운 내용을 추가한다
DEL - 현재 줄을 삭제한다

실행
RUN - 버퍼의 SQL을 실행한다
/ - SQL 퍼버의 내용을 바로 실행한다
START - SQL파일을 실행한다
@ - SQL 파일을 실행한다

환경
SET SERVEROUTPUT - 출력한 내용을 화면에 표시하도록 설정
DEFINE - 변수를 선언
ACCEPT - 사용자 입력을 받는다


PL/SQL 테스트를 위한 SQL*PLUS의 활용
ㄴ PL/SQL은 변수, 조건문, 반복문, 예외 처리등을 사용해서 여러 작업을 절차적으로 처리할 수 있다
ㄴ 단순히 SQL 실행 결과만 보는게 아니라, 내부 처리 과정이 맞는지도 확인해야 한다는 것이다.

테스트 항목
실행이 맞는가 : 문법 오류 없이 실행이 되는지 확인
출력이 맞는가 : 예상한 값이 출력이 되는지
데이터가 바뀌었는가 : Insert, Update, Delete 결과를 해야 한다 -> commit
오류가 없는가 : show Errors를 통해 오류를 확인
조건별 결과가 맞는지 : 정상값, 잘못된 값, NULL 값 등을 넣어 결과를 확인한다

DBMS_OUTPUT 패키지
ㄴ PL/SQL 처리 결과를 화면에 출력하기 위해 사용
ㄴ PUT_LINE() 메서드가 메시지 끝에 줄바꿈을 포함하여 출력
ㄴ 프로시저나 함수 사이에 단계별 출력문을 넣어서 테스트를 한다

프로시저 테스트
1. 실행전 데이터 확인
2. 프로시저 실행
3. 실행 후 데이터 확인
4. 의도대로 작동하지 않으면 디버깅

SQLPLUS SYSTEM/1111
ALTER USER HR ACCOUNT UNLOCK;
ALTER USER HR IDENTIFIED BY 1111;
CONN HR/1111

show user

SQL 처리 흐름
1. 구문 분석
ㄴ 사용자 요청한 SQL문이 데이터베이스에서 처음 사용된 문장인지, 이미 사용된 문장인지를
    공유 풀 영역을 검색하여 확인
사용된 적 있는 문장 -> 공유 풀에서 꺼내서 쓰면 됨(구문분석 필요 X)
처음 사용되었다 -> 구문분석을 실시

ㄴ 옵티마이저는 SQL문을 가장 빠르게 데이터를 검색해줄 수 있는 실행 계획을 찾는다

※ 옵티마이저
ㄴ SQL을 가장 효율적으로 실행할 방법을 결정하는 데이터베이스 내부 기능이다.

2. 실행
구문분석이 끝난 SQL을 실제로 실행하는 단계
ㄴ 서버 프로세스는 메모리 영역의 데이터베이스 버퍼 캐시 영역을 검색하여 해당 테이블의 데이터가 다른 사용자의
    다른 SQL문에 의해 존재하는지 검색

※ 데이터베이스 버퍼 캐시
ㄴ 실제 테이블 데이터를 저장하는 영역

버퍼 캐시 영역에 있으면 그대로 사용
없으면 해당 테이블의 데이터파일에서 데이터를 읽고 버퍼캐시에 저장

Insert, Update, Delete와 같은 문장이라면 데이터 버퍼 캐시 영역에서 새로운 데이터로 변경, 삭제 또는 입력을 한다

3. 추출
실행 결과를 사용자에게 보내주는 단계
서버 프로세스는 데이터베이스 버퍼 캐시 영역에서 관련 테이블 데이터를 읽어서 사용자가
요청한 클라이언트로 보내준다

SELECT문을 실행하는 경우에만 추출단계가 실행된다

SQL 작성시 고려사항
ㄴ 개발자는 SQL특성을 춘번ㄴ히 이해하여 SQL문을 적절히 구현할 수 있는 능력을 갖춰야 한다
ㄴ 사용자가 SQL 작성시, 옵티마이저가 실행 계획을 수립한 수 실행ㄷ회느 ㄴ일렬의 과정을 이해하고 작성해야 한다
ㄴ 구문분석 단계 시 옵티아미저가 수립한 실행 계획에 따라 엄ㅊ어난 수행 속도 차이가 발생할 수 있다
ㄴ 특정 SQL이 실행될 때 옵티마이저에 의해 소립된 실행 계획은 제어하기 어렵지만,
    옵티마이저가 비상식적으로 동작한다면, 이를 추적하여 개발자가 원하는 실행 계획으로
동작됟 수 있도록 조정하는 과정이 필요하다
ㄴ 개발자는 오티마이저가 정상적인 실행 계획을 수립할 수 있도록 종합적이고 전략작인
    포인트를 SQL에 부여하여 작성하여야 한다
ㄴ 좋은 SQL은, 추출되는 결과를 추론하여 SQL에 기엠 작성하여야 한다

SQL 성능 최적화를 위한 유티리리티(프로그램)의 활용
ㄴ SQL의 문제점을 파악하고 개선하여 SQL 성능을 최적화하기 위해 TKPROF 및 EXPLAIN PLAN이라는 도구를 제공하고 있다

	TKPROF
	SQL Trace 파일을 사람이 읽기 좋게 변환해주는 성능 분석 도구

	SQL TRACE
	ㄴ Oracle 데이터베이스에서 SQL이 실제로 어떻게 실행되었는지 기록하는 기능
	Parse
	ㄴ SQL 분석 단계 정보
	SQl문법 분석
	실행 계획 생성

	Execute 
	ㄴ SQL 실행 단계 정보
	ㄴ 실제 SQL 수행

	Fetch
	ㄴ 조회 결과가져오기 기능
	ㄴ SELECT 결과 반환	

	SQL TRACE 활성화
	ALTER SESSION SET SQL_TRACE = TRUE;

	SQL 실행
	SELECT * FROM EMP WHERE EMPNO = 7369;

	분석이 끝나면 비활성화
	ALTER SESSION SET SQL_TACE = FALSE;

	SQL TRACE 결과는특정 폴더에 .trc파일 형태로 저장이 된다
	TKPROF로 보기 좋게 변환하기
	결과 분석하기

	EXPLAIN PLAN의 활용
	ㄴ 사용자들이 SQL문의 액세스 경로를 확인하여성능을 개선을 할 수 있도록 SQL문을 분석하고
	    해석하여 계획을 수립하고, 관련 테이블에 저장하도록 지원해주는 도구


테스트 조작 프로시저 성능 개선하기

1. 분석가에 의해 정의된 유가사항을 기준으로, 성능 측정 도구를 활용하여 데이터 조작 프로시저의 성능을 측정
ㄴ 응답시간, 처리 건수, 리소스 사용량 등을 목표로 분석하고 테스트 대상 성능 기준을 정의한다
ㄴ 성능 도구 설정, Trace 로그 수집 준비
ㄴ 테스트용 데이터를 사용하여프로시저 실행하고, 실행결과 및 Trace 로그 수집
ㄴ 결과를 분석하여 지표를 검토한다

2. 실데이터를 기반으로 텍스트를 수행하여 데이터 조작 프로시저의 성능에 영향을 주는 병목구간을 파악한다
ㄴ 운영 데이터의 복사본 또는 유사 데이터를 준비하여 성능 테스트 환경 구성
ㄴ 대량 처리, 복합조건, 동시성 등 다양한 상황을 반영하여 테스트 시나리오 작성
ㄴ PL/SQL블록 실행 또는 SQL script 반복실행을 통해 성능 데이터 수집
ㄴ 병목 지점 식별
ㄴ 성능 저하 원인 보고서 작성

3. 테스트 결과와 정의된 요구사항을 기준으로 데이터 조작 프로시저의 성능에 따른 이슈를 해결한다
ㄴ 속도를 저하시키는 sql 구문을 재작성하여 성능을 개선한다
ㄴ 실행 계획을 개선한다
ㄴ 인덱스를 재구성한다
ㄴ 시스템 리소스 병목 구간을 조정한다
ㄴ 개선 후 동일한 테스트로 효과를 검증한다
ㄴ 성능 개선 결과를 문서화하고 공유한다
