SQL 처리 흐름
ㄴ 구문분석
	ㄴ SQL문이 사용된 적이 있는지 없는지 판단 (공유 풀 영역을 검색하여 판단)
	※ 공유 풀 영역
	ㄴ 데이터베이스 메모리 안에 있는 공간
	ㄴ 이전에 실행된 SQL문과 실행 계획이 저장될 수 있다.

	실행된 적이 있음 -> 구문분석 할 필요 없음
	실행된 적 없음  -> 구문분석 작업 실행
	ㄴ 사용자가 작성한 SQL문장을 DBMS가 이해할 수 있는 형태로 해석하는 과정
	구문분석의 3단계
	1. 문법 검사 : SQL문법이 맞는지 확인
	2. 의미 검사 : 문법은 맞지만 실제 객체가 존재하는지 확인
	3. 권한 검사 : 사용자가 해당 객체에 접근 가능한지 확인
ㄴ 실행단계
	ㄴ 구문 분석이 끝나나 SQL을 실제로 실행하는 단계
	ㄴ 구문 분석 단계에서 실행 계획이 만들어졌다면, 데이터베이스는 그 계획에 따라 실제 데이터를 찾거나 변경한다
	※실행 계획
	ㄴ 어떤 방법으로 SQL을 실행하는게 게장 빠를까? 를 계산
	ㄴ 옵티마이저 : 계산하고 실행하는 역할, 옵티마이저가 선택한 실행 방법이 실행 계획이다

	ㄴ 구문 분석이 정상적으로 끝나면 서버 프로세스는 메모리 영역의 데이터베이스 버퍼 캐시를 검사

	※ 데이터베이스 버퍼 캐시
	ㄴ 디스크에서 읽어온 데이터를 잠시 저장해두는 메모리 공간
	ㄴ 디스크는 데이터를 영구적으로 저장하지만 속도가 느리기 때문에
	ㄴ 메모리에 잠깐 저장해놓는다.
	
	데이터베이스 버퍼 캐시에 존재 -> 꺼내서 사용
	없음 -> 데이터파일에서 읽고 -> 버퍼캐시에 저장
	
	SELECT가 아닌 DML이라면 버퍼 캐시 영역에서 수정, 삭제, 추가를 한다
ㄴ 추출단계
	ㄴ SELECT에서만 실행되는 관계
	ㄴ 싫애 결과를 사용자에게 보내주는 단계

	
SQL 작성시 고려사항
1. SQL의 특성을 충분히 이해해야 한다
2. 옵티마이저가 실해애 계획을 수렵한 후, 실행되는 것을 이해해야 한다
ㄴ 이 조건은 인덱스를 사용할 수 있는가
ㄴ 이 조인은 어떤 테이블부터 읽는 것이 좋은가
ㄴ where의 조건이 너무 많지는 않은가
ㄴ 정렬이 불필요하게 발생하지 않는다?
3. 빌생계횡에 다라 수ㅐㅇ 속도가 크게 발생할 수 있다는 것을 이해해야 한다
4. 옵티마이저가 비정상적으로 동작한다면, 이를 추적하여 개발자가 원하는 실행 계획으로
   동장할 수 있도록 조정하는 과정이 필요하다
	ㄴ 인덱스 추가
	ㄴ SQL의 문수러르 변경
	ㄴ JOIN의 순서를 개선
	ㄴ 힌트

	※ 힌트 문법
	인덱스를 사용하도록 지시
	/*+ INDEX(EMP PK_EMP) */
	FULL 힌트
	/*+ FULL(EMP) */

	ㄴ 옵티마이저가 정상적인 실행 계획을 수립할 수 있도록 종합적이고 전략적인 
	    포인트를 SQL에 부여하여 작성해야 한다

6. 좋은  SQL은 결과를 추론하고 집합적으로 접근해야 한다
ㄴ 내가 최종적으로 필요한 데이터는 무엇인가?
ㄴ 몇 건 정도가 조회될 것인가?
ㄴ 어떤 조건으로 데이터를 줄일 수 있는가?
ㄴ 조인이 필요한가?
ㄴ 정렬이 꼭 필욯나가?
ㄴ 중복 제거가 필요한가?

SQL 성능 최적화를 위한 유틸리티 활용
TKPROF
ㄴ실행되는 SQL문장에 대해 분석 정보를 제공하여 사용자가 특정 SQL문장을 어떻게 
   사용해야 할 것인지에 대한 가이드라인을 제공하는 도구
EXPLAIN PLAN
ㄴ사용자들이 SQL문의 액세스 경롤르 확인낳여 성능 개선을 할 수 있도록 SQL문을
   분석하고 해석하여 실행 계획을 수립하고, 관련 테이블에 저장하도록 지원해주는 도구

데이터 조작 프로시저 성능 개선하기
업무 분석가에 의해 정의된 요구사항을 기준으로, 성능 측정 도구를 활용하여
데이터 조작 프로시저의 성능을 측정한다

1. 성능 측정을 위한 기준 요구사항 식별
ㄴ 응답 시간, 처리 건수, 리소스 사용량 등 정량적 성능 목표를 분서갛고, 테스트 대상
프로시저별 성능 기준 정의하기

2. 성능 측정 도구 설정
ㄴ 성능 도구(TKPROF 등)를 설정하고 Trace 로그 수집을 준비한다
※ Trace 파일 
	Oracle이 SQL 실행 과정을 기록한 로그 파일

3. 대상 프로시저를 실행하고 측정 데이터를 수집한다
ㄴ 테스트용 데이털르 사용하여 프로시저를 실행하고, 실행 결과 및 Trce로그를 수집한다

4. 측정 결과를 분석하여 실행 성능을 핑가한다
ㄴ 수행 시간, 블록읽기 수, CPU 사용량 등을 바탕으로 성능을 분석하고, TKPROF 결과 또는
    APM 지표를 검토한다.
※APM(Application Performance Monitoring)
ㄴ 운영중인 시스템에 대한 가용성 확보, 다운타임 초소화 등을 통해 안정적인
    시스템 운영을 위하여 부하량과 접속자 파악 등 장애 진단을 목적으로
사용하는 상능 모니터링 도구


1. 실데이터를 반영한 테스트 환경을 구성한다
ㄴ 운영데이터의 복사본 또는 유사 데이터를 준비하여 성능 테스트 환경을 구성한다

2. 테스트 시나리오를 설계한다
ㄴ 대량 처리, 복잡 조건, 동시성 등 다양한 상황을 반영하여 테스트 시나리오를 작성한다

3. 테스트를 수행하고 성능 로그를 수집한다
ㄴ PL/SQL 블록 실행 또는 SQL Script 반복 실행을 통해 성능 데이터를 수집한다

4. 병목 지점을 식별
ㄴ 실해애 계획을 분석하여 인덱스 미사용, Full Scan, 테이블 조인 방식 등을 점검하고,
통계 정보 최신화 여부를 확인한다.

5. 성능 저하 원인을 문서화 한다
ㄴ 병목이 발생한 SQL과 원인을 정하하고, 필요 항목을 도출하엽누석 보고소를 작성한다

----------------------------------------------------------------------------------------------------

테스트 결과와 정의된 요구사항을 기준으로 데이터 조작 프로시저의 성능에 따른 이슈를 해결한다

1. 속도를 저하시키는 SQL 구문을 재작성하여 성능을 개선한다
ㄴ WHERE 절 최적화, 서브쿼리 정리, IN 대신 EXIST 사용 등을 통해 쿼리 성능을 향상한다

2. 실행 계획 개선
ㄴ 옵티마이저 힌트를 적용하거나, Driving 테이블을 변경하여 최적의 실행 경로로 유도한다
※ Driving 테이블
	조인을 수행할 때 가장 먼저 있는 테이블

3. 인덱스를 재구성한다
ㄴ 새로운 인덱스를 추가하가너ㅏ 기존 인덱스를 조정하여 접근 효율성을 높인다

4. 시스템 리소스 병목 구간을 조정한다
ㄴ Shared Memory, CPU, I/O 등 인프라 병목 구간이 원인일 경우 시스템 관리자의 협조를 통해 시스템 설정을 최적화한다.

5. 개선 후 동일한 테스트로 효과를 검증한다
ㄴ 동일한 테스트를 반복 실행하여 개선 전후 성능 차이를 수치로 비교하고 효과를 검증한다

6. 성능 개선 결과를 무넛화하고 공유한다
ㄴ 개선 전후 비교, 변경 내역, 테스트 결과를 정리하여 성능 개선 보고서를 작성한다

논리 데이터 모델의 구성 요소 중, 데이터베이스에 명시된 규정에 따라 릴레이션이 가질 수 있는 유효한 데이터 값의
범위를 제약하는 규칙을 의미하는 용어를 작성하시오 ( 개체 무결성, 참조 무결성 등)
>> 제약조건

관계 데이터베이스 모델에서 상위 테이블의 기본키(Primary Key)를 참조하여 테이블 간의 연결 고리 역할을 하는
속성을 무엇이라고 하는지 작성하시오
>> 외래 키 

논리 데이터 모델링의 주요 목적 중 하나는 데이터의 중복을 제거하여 이상 현상을 방지하는 것이다.
데이터베이스에서 발생할 수 있는 이상 현상의 3가지 종류를 제시하고, 각각의 개념을 설명하시오
>> 삽입이상, 삭제이상, 갱신이상

데이터베이스 설계 과정에서 개체 무결성과 참조 무결성은 데이터의 정확성을 보장하는 핵심 규칙이다.
이 두 제약 조건의 개념과 위배되지 않기 위한 세부 규칙을 비교하여 서술하시오
>> 개체 무결성 : Null 값이 들어가면 안되고, 중복되는 값이 들어가면 안된다
>> 참조 무결성 : 외래키 값은 참조 대사잉 되는 관계 릴레이션의 기본키 값과 동일하거나, Null이 되어야 한다

논리 데이터 저장소 확인 단계에서는 관계형 데이터 모델의 설게 품질을 높이기 위해 정규화 과정을 검토한다
제 1정규형부터 제3 정규형까지의 정규화 과정을 각 단계별 변환 조건을 포함하여 단계적으로 서술하시오
>> 1 정규형 : 모든 테이블 값이 원자값이어야 한다
>> 2 정규형 : 1 정규형을 만족하면서, 기본키가 아닌 모든 속성이 기본키에 완전함수적 종속을 만족해야 한다
>> 3 정규형 : 2차 정규형을 만족하고, 기본키가아닌 모든 속성이 이행함수적 종속을 만족하지 않아야 한다

관계형 데이터 모델에서 릴레이션의 행을 뜻하는 용어로, 카디널리티의 수와 직결과는 이것은 무엇인가?
>> 튜플

개념적 데이터 모델링 단계에서 도출된 ERD를 관계형 데이터베이스의 논리적 테이블 구조로 변환하는 과정을
릴레이션 스키마 변환이라고 한다. 이 과정에서 1:N 관계를 가진 두 개체가 테이블로 변환될 때 외래키가 지정되는 규칙은?
>> 1:N 관계의 테이블을 1:1 테이블로 바꾼다

데이터베이스의 정규화 과정은 데이터 무결성을 높여주지만, 과도한 정규화는 시스템의 성능 저하를 유발할 수 있다
정규화가 조회 성능을 떨어뜨리는 이유를 연산의 관점에서 서술하시오
>> 

제2정규형의 정의를 설명하고, 테이블이 제1정규형은 만족하지마 제2정규형을 만족하지
못할 때 발생하는 문제를 복합키 관점에서 서술하시오
