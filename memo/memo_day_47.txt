ㄴ 엔티티, 속성, 관계
ㄴ 정규화

물리 데이터 저장소 설계하기
ㄴ 반정규화 : 테이블이 너무 나눠져 있어서 JOIN을 너무 많이해서 오히려 조회성능이 떨어져서
	중복테이블 추가
	ㄴ 집계 테이블 추가
	ㄴ 상태 컬럼을 담고 있는 테이블을 생성
	ㄴ 특정 부분만을 포함하는 테이블 추가

	테이블 조합
	ㄴ 항상 같이 사용되는 두 개 이상의 테이블을 하나로 합치는 것

	테이블 분할
	ㄴ 수직분할 : 컬럼별 사용 빈도의 차이가 많은 경우 자주 사용되는 컬럼과 
		그렇지 않는 컬럼으로 분류하여 테이블 분할
	ㄴ 수평분할 : 특정 범위별 사용 신보 차이가 많은 경우 해당 범위별로 테이블을
		분할하는 방법

	테이블 제거
	ㄴ 테이블 재정의나 컬럼의 중복화로 더 이상 사용되지 않는 테이블을 발생하면 삭제하기

	컬럼중복
	ㄴ 필요한 해당 테이블이나 컬럼을 중복하여 추가
	ㄴ 다른 테이블에 있는 컬럼을 현재 테이블에도 추가로 저장하는 것

--------------------------------------------------------------------------------

물리 데이터 모델링 하기
ㄴ 논리 모델링을 통해 얻은 내용을 데이터베이스에 사용할 수 있는 물리데이터 모델로 변환
	엔티티 -> 테이블로 변환
	ㄴ 일반적으로 테이블과 엔티티 명칭을 동일하게 하는 것을 권고
	속성 -> 컬럼으로 변환
	ㄴ 속성의 명칭과 반드시 일치할 필요는 없으나, 개발자와 사용자 간의 의사소통을
	    위하여 가능한 표준화된 언어를 사용하도록 한다
	ㄴ SQL 예약어 사용은 피해야 한다
	ㄴ SQL 문장의 가독성을 높이기 위해 컬럼 명칭은 가능한 짧은 것이 좋다
	UID -> 기본키로 변환
	관계 -> 외래키로 변환
	ㄴ 외래키로 설정된 컬럼의 이름도 보통은 기본키 쪽의 이름을 그대로 사용
	ㄴ 다른 의미를 가지면 바꿔도 된다
	컬럼 타입 및 길이를 정의
	ㄴ 내가 넣으려는 데이터의 크기를 파악해서 컬럼의 길이를 저장
	반정규화 적용 여부를 검토

물리 데이터 저장소 구성

테이블 제약조건
PK ,FK, CHECK, DEFAULT, NOT NULL, UNIQUE

	삭제할 때
	ㄴ 참조된 기본키의 값이 삭제될 경우 어떻게 처리할지 정하는 규칙
	ㄴ 고객이 삭제되면, 그 고객의 주문 데이터는 어떻게 할 것인지

	CASCADE
	ㄴ 부모테이블의 데이터가 삭제되면, 그 데이터를 참조하고 있던 
	    자식 테이블의 데이터도 함께 삭제되는 방식

인덱스 설계
ㄴ 데이터를 빠르게 찾기 위해 만드는 검색용 목차

ㄴ 단순히 값의 종류만 보지말고, 실제 조회 조건에서 몇 건이 조회되는지 봐야 한다

수정이 빈번하지 않은 컬럼을 선정한다
ㄴ 인덱스는 조회를 빠르게 해주지만, 삽입, 수정, 삭제 작업에는 부담이 된다
ㄴ 인덱스로선정한 커럶의 정리 내용을 따로 저장하기 때문에 테이블 데이터가
    바뀔 때 인덱스도 같이 바꿔어야 한다

설계시 고려사항
ㄴ 새 인덱스가 기존 액세스 경로에 영향을 줄 수 있다
	ㄴ 엑서스 경로 : DBMS가 데이터를 찾기 위해 선택한 이동방법

ㄴ 대표적인 액세스 경로
	1. Full Table Scan(전체 테이블 검색)
	ㄴ DB가 처음부터 끝까지 읽는다
	2. Index Scan(인덱스 검색)
	ㄴ 인덱스를 사용해서 필요한 위치만 바로 찾는다

지나치게 많이 만들면 오버헤드가 된다
오버헤드 : 빠르게 찾기 위해 만들어둔 목차가 너무 많아서, 오히려 관리가 힘들어지는 것

인덱스는 추가 저장공간이 필요하다
ㄴ 인덱스는 테이블과 별도 로 저장되는 검색구조이다.
ㄴ 인덱스를 만들면 저장 공간을 추가로 확보한다

넓은 범위를 인덱스로 처리하면 오히려 느릴 수 있다
ㄴ 적은 데이터를 빠르겣 찾을 때 효과적이다
ㄴ 하지만 전체 데이터가 많은 부분을 조회하려면 오히려 전체 테이블을
    읽는 것이 더 빠를 수 있다.

인덱스와 테이블 데이터의 저장 공간을 적절히 분리한다
ㄴ 대규모 DB에서는 테이블 데이터와 인덱ㄷ스 데이터를 물리적으로분리해서 자장하기도 한다

뷰(View) 설계
ㄴ SELECT한 결과를 테이블인 것 처럼 저장하는 가상 테이블
	ㄴ 복잡한 SQL을 단순하게 사용할 수 있다
	ㄴ 특정 컬럼만 보여줄 수 있다
	ㄴ 조건에 맞는 데티어만 보여줄 수 있다

뷰 설계시 고려사항
ㄴ 뷰는 결국 테이블을 조회한단
	ㄴ 실제 데이터를 저장하기 않는다
	ㄴ 뷰는 내부적으로 SELECT문을 실행해서 결과를 보여준다
	ㄴ 그래서 뷰를 사용한다고해서 무조건 빨라지는 건 아니다
	ㄴ 오히여 뷰 안의 SQL이 복잡하면 성능 문제가 생길 수 있다 
	
	복잡한 뷰는 성능이 느려질 수 있다
	뷰 안의 SELECT문의 조건도 인덱스를 고려해야 한다

파티션
ㄴ 하나의 큰 테이브릉ㄹ 내부적으로 여러 조각으로 나누어 저장하는 기술
ㄴ 겉으로는 하나의 테이블로 보여진다

왜 파티션이 필요한가?
ㄴ 대용량 데이터를 여러 영역으로 나누어 저장함으로써, 조회시 필요한 데이터만 읽어 성능을 향상시키고,
    백업, 복구, 삭제 같은 관리 작업을 효율적으로 수행하기 위해 사용한다

파티션
목적 : 데이터를 나눠서 관리
개념 : 읽을 데이터 범위를 축소
데이터를 여러 영역으로 분리 저장
창고를 여러개 나누는 것
필요한 영역만 읽어서 속도를 향상

파티션의 종류
범위 분할
ㄴ 특정 컬럼 값의 범위에 따라 데이터를나누어 저장하는 방법

--------------------------------------------------------------------------------

해시 분할의 장점
데이터 분산 : 특정 파티션에 데이터가 물리는 현상을 감소시킨다
저장 균형 ; 여러 파티션 크기를 비슷하게 유지
특정 키 조회 빠름 : ID 기본 조회에 적합
대용량 처리 : 부하를 여러 파티션으로 분산할 수 있다

조합 분할(Composite partitioning)
ㄴ 파티션 기준을 2개 이상 좋바해서 데이터를 나누는 방식
ㄴ 먼저 큰 기준으로 한 번 나누고, 그 안에서 다시 작은 기준으로 한 번 더 나누는 방식

--------------------------------------------------------------------------------

클러스터 설계
ㄴ 자주 조회되는 데이터를 디스크의 가까운 위치에 모아 저장해서 조회 속도를 높이는
    물리적 저장방법
	
	일반 저장 방식 : 책이 아무 책장이나 흩어져 있는 상태
	클러스터 저장 방식 : 자주 같이 보는 책들은 같으 책장이나 가까운 칸에 모아두는 방식
	그렇기 때문에 조회할 때는 빠르다.
	하지만 책을 새로 넣거나 위치를 바꾸거나 삭제할 때는 정리 비용이 더 소모된다

	파티션 : 저장공간을 나눈다.
	클러스터 : 관련 데이터를 가까이 저장한다.


단일 테이블 클러스트링과 다중 테이블 클러스트링을 구분한다
단일 테이블 클러스트링
ㄴ 하나의 테이블 안에서 특정 기준으로 데이터를 모아 저장하는 방식
