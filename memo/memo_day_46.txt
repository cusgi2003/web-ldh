데이터 입출력 구현 능력단위
논리 데이터저장소 확인하기
ㄴ 엔티티, 속성, 관계가 잘 설정됐는지
ㄴ E-R 다이어그램 잘 그렸는지
ㄴ 정규화 잘 됐는지( 1~5차, 함수적 종속성, 이상현상 )

--------------------------------------------------------------------------------

논리 데이터모델 검증하기
ㄴ 필요한 데이터가 빠지지 않았는지(입력, 출력, 연계 데이터)
ㄴ 현재 시스템의 데이터구조를 파악했는지(기본 DB구조, 백업, 공통코드, 외부 연계 데이터)
ㄴ 사용자 요구사항이 반영되었는가
ㄴ 데이터 흐름이 명확한가
ㄴ 데이터 모델이 적절한다
ㄴ 접근권한과 통제가 명확한가

물리 데이터저장소 설계하기
ㄴ 논리 데이터저장소 설계를 바탕으로 목표 시스템의 데이터 특성을 반영하여
    최적화된 물리 데이터저장소 설계

--------------------------------------------------------------------------------

반 정규화
ㄴ 정규화를 조금 포기하고, SELECT 성능을 높이기 위해 데이터의 중복을 감수하고,
    테이블 구조를 바꾸는 설계 방법

반정규화를 하는 이유
ㄴ 테이블이 너무 잘게 쪼개져 있어서 JOIN을 너무 많이 해야해서
ㄴ 데이터가 적을때는 괜찮은데 데이터가 많아지면 SELECT 속도가 느려진다
ㄴ 자주 조회하는 데이터를 일부로 합치거나 중복해서 저장한다

핵심개념
ㄴ JOIN을 줄여 빠르게 조회속도 향상
ㄴ 합계, 평균을 저장하여 계산 비용 감소
ㄴ 자주쓰는 속성은 테이블로 따로 만들어 분리
ㄴ 항상 같이 쓰는 테이블을 하나로 합친다.

반정규화 유형

1. 중복테이블 추가
ㄴ 기존 테이블과 비슷한 데이터를 가진 테이블을 추가로 만드는 것
ㄴ 주로 조회속도를 높이기 위해 사용

집계테이블 추가
ㄴ 합계, 평균, 개수같은 계산 결과를 미리 저장해두는 테이블

진행테이블 추가
ㄴ 현재 처리중인 업무 상태를 빠르게 확인하기 위해 만드는 테이블
ㄴ 진행중인 데이터만 저장하는 테이블을 생성

특정 부분만 포함하는 테이블 추가
ㄴ 큰 테이블에서 자주 사용하는 일부 컬럼만 빼서 테이블을 만드는 방법

테이블 조합
ㄴ 항상 같이 사용되는 두 개 이상의 테이블을 하나로 합치는 것

테이블 분할
ㄴ 하나의 테이블을 여러개로 나누는 것
	ㄴ 수직분할
		ㄴ 컬럼별로 사용 빈도의 차이가 많이 나는 경우 자주 사용되는 컬럼들과 그렇지 않는 컬럼으로 
		    분류하여 테이블을 분할하는 방법
		ㄴ 자주 쓰는 컬럼만 빠르게 조회
		ㄴ 이미지, 긴 글같은 무거운 데이터가 있을 때
		ㄴ 개인정보 접근 권한을 분리해야 할 때
	ㄴ 수평분할
		ㄴ 특정 범위별 사용 빈도의 차이가 많은 경우 해당 범위별로 테이블을 분할하는 방법
		ㄴ 행을 기준으로 테이블을 나누는 것
		ㄴ 사용 상황
			ㄴ 날짜별 조회가 많음
			ㄴ 지역별 데이터가 많음
			ㄴ 상태별 조회가 많다

테이블 제거
ㄴ 테이블 재정의나 컬럼의 중복화로 더 이상 사용되지 않는 테이블이 발생할 경우, 
    해당 테이블은 삭제한다

컬럼 중복
ㄴ 필요한 해당 테이블이나 컬럼을 중복하여 추가한다
ㄴ 다른  테이블에 있는 컬럼을 현재 테이블에도 추가로 저장하는 것이다.

--------------------------------------------------------------------------------

지나치게 많은 인덱스는 오버헤드가 된다
오버헤드 : 추가 부담
ㄴ 인덱스가 많으면 조회는  빠랄질 수 있지만, 변경작업은 느리다
ㄴ 예를 들어 회원 1명을 추가할 떄 테이블에만 저장하는 것이 아니라 여러 인덱스도 함께 갱신해야 한다
	-> 인덱스가 5개라면, 회원 데이터 1건을 넣을 때 인덱스도 5개도 함께 관리해야 한다
인덱스는 추가 저장공간이 필요하다
ㄴ 테이블과 ㅂ려도로 저장되는 검색 구조이다.
ㄴ 인덱스를  만들면 저장 공간을 추가로 사용한다
ㄴ 그래서 무조건 많이 만드는 것이 좋지 않다

넓은 범위 인덱스로 처리하면 오히려 느릴 수 있다.
ㄴ 인덱스는 적은 데이터를 빠르게 찾을 때 효과적이다
ㄴ 하지만 전체 데이터의 많은 부분을 조회하면 오히려 전체 테이블을 읽는 것이 더 빠를 수 있다.

인덱스와 테이블 데이터의 저장 공간을 적절히 분리한다
ㄴ 대규모 DB에서는 테이블 데이터와 인덱스 데이터를 물리적으로 분리해서 저장하기도 한다
ㄴ 이렇게 하면 디스크 입출력 성능이나 관리측면에서 유리할 수 있다

뷰(View)
ㄴ 실제 데이터를 저장하는 테이블이 아니라, SELECT문을 저장해 둔 가상의 테이블
ㄴ 자주 사용하는 조회 SQL에 이름을 붙여 둔 것

뷰가 필요한 이유
	ㄴ 복잡한 SQL을 단순하게 사용할 수 있다
	ㄴ 여러 테이블을 JOIN하는 SQL은 길고 복잡하다
	ㄴ 특정 컬럼만 보여줄 수 있다
		ㄴ 개인정보, 급여와 같은 민감한 정보는 제외하여 뷰를 만들면 필요한 컬럼만 공개할 수 있다.
	ㄴ 조건에 맞는 데이터만 보여줄 수 있다
		ㄴ 활성 회원만 보여주는 뷰
	
뷰 속성
REPLACE
	ㄴ 뷰가 이미 존재하는 경우 기존 뷰를 새 내용으로 다시 생성하는 속성
FORCE
	ㄴ 기존테이블이 존재하지 않아도 뷰를 생성하는 속성
	ㄴ 테이블이 없어도 View 정의 자체는 만들수 있지만, 조회하려고 하면 오류가 발생할 수 있다.
NOFORCE
	ㄴ 기본 테이블이 존재할 때만 뷰를 생성하는 속성
	ㄴ ORACLE에서 기본값은 보통 NOFORCE이다.
	ㄴ 일반적으로는 참조할 테이블이 있어야 뷰를 만들 수 있다.
WITH CHECK OPTION
	ㄴ 뷰의 조건을 만족하는 행만 INSERT 또는 UDPATE할 수 있도록 제한하는 속성
	ㄴ 이 뷰의 조건을 벗어나는 수정은 허용하지 않는다.
WITH READ ONLY
	ㄴ 뷰를 조회 전용으로 만드는 속성

뷰 설계시 고려사항
뷰는 결국 테이블을 조회한다
ㄴ 실제 데이터는 저장하지않는다.
ㄴ 실제로는 내부에서 원본 테이브르이 SELECT가 실행되는 것과 같다
ㄴ 그래서 뷰를 사용한다고 해서 무조건 빨라지는 것은 아니다.
ㄴ 오히려 뷰 안의 SQL이 복잡하면 성능 문제가 생길 수 있다

복잡한 뷰는 성능이 느려질 수 있다
ㄴ 뷰를 조회하게 되면 실제로는 member와 orders 테이블을 조인하고  집계한다
ㄴ 데이터가 많으면 이 작업은 느려질 수 있다.

뷰 안의 select 조건도 인덱스를 고려해야 한다
ㄴ 뷰 안에 조건이 있다면, 해당 조건 컬럼에 인덱스가 있는지 확인해야 한다.

뷰를 사용하는 SQL의 WHERE절도 중요하다
ㄴ 뷰 안의 조건이 부족하더라도, 뷰를 조회하는 SQL에서 좋은 조건을 줄 수 있다.


파티션
ㄴ 큰 테이블을 여러 개의 작은 영역으로 나누어 저장하는 방법
ㄴ 만약 모든 데이터를 하나의 테이블에 저장하면 검색할 때 길이가 엄청난 데이터를 확인해야 한다
	ㄴ 시간이 오래 걸리기 때문에 데이터를 나눈다.

파티션이 필요한 이유
ㄴ 대용랑 시스템에서는 데이터가 계속 증가한다
ㄴ 데이터가 많아지면
	ㄴ 조회 속도저하
	ㄴ 백업 시간 증가
	ㄴ 복구 시간 증가
	ㄴ 디스크 사용량 증가

파티션 종류
	1. 범위 분할
		ㄴ 지정한 컬럼 값의 범위를 기준으로 데이터를 나누는 방법

		반정규화 분할 	-> 테이블을 진짜로 여러개로 만듦
		파티션 		-> 하나의 테이블 내부에서 저장영역을 나눔

		ㄴ 파티션을 만들어도 사용자는 일반 테이블처럼 사용한다
		ㄴ 원래 하던 것처럼 CRUD 작업을 한다
