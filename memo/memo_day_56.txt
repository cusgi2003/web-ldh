너무 세분화하지 않는다. ( 버튼 하나하나까지 표현하면 너무 복잡하다 )

2. UI 화면 설계의 기본 구서 ㅇ요소
ISO 9241 161에 항목들이 정의가 되어있다.
컨테이너, 버튼, 체크박스, 라디오 버튼, 입력창, 라벨, 목록, 대화상자, 폼, ...

3. 유용성 개념 및 적용 원리 파악
- 유용성
	사용자가 시스템을 활용하여 자신이 원하는 목표를 얼마나 효과적이고
	효율적으로 달성할 수 있는지 나타내는 척도
- 유용한 시스템을 설계할 때 고려사항
	사용자가 머리속에서 그리는 구조, 기능, 가치에 대한 이해와 실제로 설계자가 구현하려는
	모형 사이의 차이를 줄이려는 노력이 필요하다
- 사용자 모형과 개발자 모형간의 차이 발생
	시스템에서 제공하는 실행 가느한 기능과 사용자가 본래 기대하는 목적이 서로 다를 때 발생
	시스템이 실행된 결과가 사용자의 의도와 일치하지 않을 때 발생
3-1. 실행 차를 최소화하기 위한 UI 설계 원리
※ 실행 차 : 사용자가 달성하고자 하는 목적과, 실제로 수행하기 위히 시스템에 제공되는 기능 간의 간극
- 사용 의도 파악
- 행위 순서 규정
ㄴ 사용자가 행동 과정을 세분화해서 순서대로 명확하게 안내
ㄴ 한 가지 작업을 완료하는 단계는 가능한 최소화 해야 한다.
ㄴ 행위 순서는 사용자의 기존 경험과 자연스럽게 이어지도록 친숙하게 구성하는 것이 좋다
- 행위의 순서대로 실행
- 사용자가 계획한 작업 절차를 UI 상에서 직관적으로 파악할 수 있도록 제공하여,
의도한 행위를 실제로 실행하는 과정에서 혼란이나 어려움이 생기지 않도록 해야 한다.
ㄴ 불필요하게 많은 상호작용으로 인해 작업 진행이 방해받지 않도록 주의해야 한다.
ㄴ 사용자가 의도한 행위에 가장 적합한 입력 장치를 선택한다
ㄴ 적절한 피드백과 취소 기능을 제공
- 기본 값을 알맞게 설정해 불필요한 조작을 줄여야 한다
3-2. 평가 차를 줄이기 위한 UI 설계 원칙
※ 사용자가 의도한 목적과 시스템이 제공한 실행 결과가 서로 상이하여 발생하는 격차
- 키 입력 결과를 사용자가 신속하게 인식할 수 있도록 설계
ㄴ 사용자가 어떤 조작을 해도 화면에 아무런 변화가 없다면, 사용자가 제대로 입력했는지 의심하게 된다.
	가능한 빠르게 시스템이 반응하도록 설계 해야되고, 즉각적인 반응이 어렵더라도 지연을 최소화해야 한다.
- UI 조작으로 인해 시스템의 상태 변화를 쉽게 ㅔ파악할 수 있도록 설계
ㄴ 사용자의 입력에 따라 시스템이 어떻게 달라졌는지 명확히 인식할 수 있어야 한다.
- 사용자의 의도와 시스템의 반응 결과의 일치 여부를 쉽게 확인할 수 있도록 설계
ㄴ 사용자가 기대한 결과가 실제 시스템을 통해 충족되었는지 쉽게 알 수 있어야 한다.
	이를 위해 미리보기와 같은 기능을 제공하면 사용자가 예상 결과를 사전에 확인할 수 있다.


4. 스토리보드 작성 방법
스토리보드
- 서비스 구축 과정에서 디자이너와 개발자가 공통으로 참고하는 최종 산출물 정책, 프로세스, 콘텐츠 구성, 와이어 프레임,
	기능 정의, 데이터베이스 연동 등 서비스 구현에 필요한 핵심 정보가 담긴다.
- 어떤 메뉴가 있고, 어떤 순서로 배치되며, 각 메뉴를 뭐라고 부를지 정의
4-1. 메뉴 구성도 작성
- 전체 메뉴의 흐름을 정리하고, 어떤 내용을 보여 줄지를 확인하여 메뉴 구성에 반영한다

4-2. 스타일 확정
- 레이아웃, 글꼴 모양과 크기, 색상, 그래픽 요소 등을 통일감 있게 유지한다.
- 기준이 없으면 화면마다 제각각이 된다.

4-3. 설계
- 실제 화면에 표현될 시각적 디자인 콘셉트를 구체화 한다

데이터베이스 연동
- 화면에서 사용자가 한 행동이 데이터베이스의 어떠 ㄴ데이터와 연결되는지 설명하는 것



UI 시나리오 작성 원칙
1. 사용자 상호작용 중심 설계
2. 적용 수준 구분
3. 접근성과 포용성 반영
4, 상호작용, 피드백, 오류 처리
5. 공통 요소와 반복 패턴 활용
6. 실제 사용자 데이터 기반 구성
7. 화면 사용 도움 정보 포함



1. UI 시나리오 문서 목적 및 대상 사용자
목적 : 핵시 ㅁ사용자 흐름과 과업 중심 상호작용을 명확하게 기술하여, 실제 서비스 구현 시 혼선이 발생하지않도록 한다.
대상 : 화면 설계에 참여하는 모든 인원 ( 기회갖, 디자이너, 퍼블리셔, 개발자 등)

2. 구성 요소
- UI 시나리오 문서에는 
개요 : 사용자 목표, 시나리오 목적, 기본 전제 조건 제시
사용 흐름 : 단계별 화면 이동, 상태 변화, 화면 전환 로직
UI 요소 구성 : 아이디어 도출 미 ㅊ설계 방향성 탐색
상호작용 및 입력 로직 : 버튼 클릭, 입력 필드, 오류 상황대응, 피드백 흐름 정의
접근성 고려사항 : 웹 접근성 기준 준수, 키보드 제어, 명확한 라벨과 판독성 확보, 오류 발생시 복구 절차,
사용자 안내 메시지 설정
예외 및 오류 시나리오 : 오류 발생 시 흐름, 복구 절차, 사용자 안내 메시지 설계
적용 수준 : 필수, 권장, 우수 수준으로 구분하여 단계적 적용 계획 수립이 포함되어야 한다.

3. 작성 원칙
- 일관된 표현의 사용
디자인 원칙, 스타일 가이드에 따른 컴포넌트 명칭과 용어를 일관되게 사용하기

버튼 ( 확인, 완료 저장하기 ) -> 저장
사용자 ( 회원, 유저, 사용자 ) -> 사용자
로그인 ID (계정, ID, 아이디) -> 아이디
오류(에러, 오류, 실패) -> 오류

- 사용자 중심
실제 사용자가 수행하는 단계를 중심으로 작성해야 한다

- 현장 적용 가능
UI 요소의 위치, 텍스트 라벨, 상호작용 과정의 상태값 들을 구체적으로 명히사여 설계와 구현 단계에서 바로 활용할 수 있도록 한다.

4, 시나리오와 서비스 패턴 연계
- UI 시나리오는 서비스에서 자주 사용되는 패턴과 연결해서 작성해야 한다.
EX ) 로그인 패턴, 검색 패턴, 신청 패턴, 동의 패턴 
- 각 단계에서 필수, 권장, 우수 수준의 적용 사항을 구분해 기술하며, 차후 개선 계획 수립에도 활용될 수 있어야 한다.

5. 접근성 및 포용성
- 웹 접근성 기준인 CSAG 2.1 수준을 충족해야 한다.
라벨 제공, 색상 대비 확보, 키보드 내비게이션 지원, 오류 메시지 설계
- 모든 사용자를 고려한 포용적 디자인 관점에서 각 화면 흐름을 검토한다
