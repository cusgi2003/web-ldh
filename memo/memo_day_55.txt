2. UI 화면 설계의 기분 요구사항
ISO 9241-161
ㄴ 사용자와 컴퓨터의 상호작용에 대한 국제 표준
ㄴ 화면 설계시 자주 사용되는 기본 UI 구성요소를 점검하고 있다.
ㄴ 그 중 기본 구성 요소를 버튼, 체크박스, 입략창 등 51개를 항목으로 제시한다.
주요 UI 구성 요소
ㄴ 컨테이너	(다른 요소드릉ㄹ 포함하는 요소)
ㄴ 버튼		(명령 실행, 제출, 다른 페이지이동)
ㄴ 체크박스	(복수 선택 가능한 선택 목록)
ㄴ 라디오버튼	(그룹 중 하나만 선택)
ㄴ 입력창		( 사용자 텍스트 일력, 검색 조건 입력)
ㄴ 라벨		(제어 요소의 식별자 및  설명을 제공)
ㄴ 목록		(항목 정렬)
ㄴ 대화상자	(사용자 피드백, 확인 요청, 알림 등)
ㄴ 폼			(입력, 선택, 클릭과 같은 전체를 구성하는 복합 요소)


3. 유용성 개념 및 적용 원리 파악

유용성
ㄴ 사용자가 시스템을 활용하여 자신이 원하는 목표를 얼마나 효과적이고 효율적으로
    달성할 수 있는지를 의미한다.
유용성이 높은 시스템
로그인 버튼을 쉽게 찾을 수 있다
원하는 메뉴를 바로 찾을 수 있다
검색 결과가 빠르게 나온다

유용성이 낮은 시스템
메뉴 위치를 찾기 어렵다
버튼 이름이 이해되지 않는다
기능을 사용하기 위해 너무 많은 단계를 거쳐야 한다

3-2. 유용한 시스템을 설계할 때 고려사항
ㄴ 시스템을 설계할 때 사용자가 생각하는 방식과 실제 시스템이 동작하는 방식의 차이를 줄여야 한다
	회원가입 -> 약관 설명 화면 -> 광고화면 -> 회원가입 화면 방식으로 동작하면 사용자는 불편함을느낌
	사용자가 예상하는 동작과 실제 동작이 다르면 혼란이 발생하고, 오류가 증가하고, 사용시간이 증가하고,
	서비스 만족도가감소할 수 있다

3-3 사용자 모형과 개발자 모형간의 차이 발생
사용자 모형(User Model)
ㄴ 사용자가 머릿속으로 생각하는 시스템의 모습
개발자 모형(Developer Model)
ㄴ 개발자가 시스템을 구현한 방식

시스템에서 제공하는 실행 가능한 기능과 사용자가 본랜 기대한 목적이 서로 다를 때 발생

3-4. 실행 차를 최소화하기 위한 UI 설계 원인
※ 실행 차 : 사용자가 원하는 행동 실제 시스템 조작 사이의 차이

사용 의도 파악
ㄴ 사용자가 무엇을 이루려고 하는지 명확하게 이해
ㄴ 불필요한 부가 기능으로 인해 시스템 성능이 저하되지 않도록 설계
ㄴ 사용자가 관심을 가질만한 항목은 눈에 잘 띄는 곳에 두기
ㄴ 필요한 순간에 기능이 제공되도록 배치

행위 순서 규정
ㄴ 사용자의 행동 과정을 세분화하여 순서대로 명확하게 안내하되, 필요한 경우
    사용자가 의도에 맞게 순서를 자유롭게 바꿀 수 있다.
ㄴ 한 가지 작업을 완료하는 단계는 가능한 최소화 하기
ㄴ 동일한 결과를 다양한 방식으로 얻을 수 있도록 설계에 반영하기
ㄴ 행위 순서는 사용자의 기존 경험과 자연스럽게 이어지도록 친숙하게 구성하기

행위의 순서대로 실행
ㄴ 사용자가 계획한 작업 절차를 UI 상에서 직관적으로 파악할 수 있도록 제공하여 의도한 행위를
    실제로 실행하는 과정에서 혼란이나 어려움이 생기지 않도록 해야한다
ㄴ 불필요하게 많은 상호작용으로 인해 작업 진행이 방해받지 않도록 주의해야 한다.
ㄴ 기본값을 알맞게 설정해 불필요한 조작을 줄여야 한다.

3-3. 평가 차를 줄이기 위한 UI 설계 원칙
※ 평가 차 : 사용자가 시스템의 상태를 이해하는 과정에서 발생하는 차이

키 입력 결과를 사용자가 신속하게 인식할 수 있도록 설계
ㄴ 사용자가 어떤 조작을 했음에도 화면에 아무런 변화가 없다면, 사용자가 제대로 입력했는지
	의심하게 되고 이는 평가 차리를 키우게 된다
ㄴ 따라서, 가능한 빠르게 시스템이 반응하도록 설계해야 하며, 즉각적인 반응이 
	어렵더라도 지연을 최소화해야한다
ㄴ 또한 사용자가 행위를 했다는 사실이 명확하게 들어나도록 직관적인 피드백 제공하기

UI 조작으로 인해 시스템의 상태 변화를 쉽게 파악할 수 있도록 설계
ㄴ 사용자의 입력에 따라 시스템이 어떻게 달라졌는지 명확하게 인식할 수 있어야 한다
ㄴ 이를 위해 상태 정보를 단순하고, 이해하기 쉽게 표현해야 하며, 복잡하지 않도록 설계하는 것이 좋다.

사용자의 의도와 시스템 반응 결과의 일치 여부를 쉽게 확인할 수 있도록 설계
ㄴ 사용자가 기대한 결과가 실제 시스템을 통해 충족되었는지 쉽게 알 수 있어야 한다
ㄴ 이를 위해 미리보기와 같은 기능을 제공하면 사용자가 예상 결과를 사전에 확인할 수 
	있어 대부분의 경우 만족도를 높일 수 있다.



4. 스토리보드 작성 방법
ㄴ 스토리보드는 서비스 구축 과정에서 디자이너와 개발자가 공통으로 참고하는 최종 산출물이다.

	서비스 구현에 필요한 핵심 정보가 들어가 있다.
ㄴ 화면 설계와 기능 정의를 문서화 하여 개발 과정의 기준이 된다


4-1. 메뉴 구성도 작성

ㄴ 화면 간 이동 관계를 명확하게 정리한다.

4-2. 스타일 확정
ㄴ 서비스 전체의 디자인 일관성 유지하기
ㄴ 화면마다 다른 디자인이 적용되지 않도록 기준 설정하기
	색상, 글꼴, 버튼 스타일, 입력창 스타일, 아이콘 및 그래픽

4-3. 설계

ㄴ 입력창(input)을 확인하고, 규칙을 정의한다
	ㄴ 입력 가능 여부를 구분하여 CSS(스타일)를 적용한다
	ㄴ 필드의 길이는 자료의 길이와 일치하게 만든다
	ㄴ 텍스트 정렬 방식은 가운데 정렬한다
	ㄴ 라벨은 좌측 정렬한다.
	ㄴ 필수 입력 항목을 구분해 Label에 다음과 같이 CSS를 적용하기.
일반 입력 Label : HTML <th> </th>
필수 입력 Label : Text 뒤에 * 표시를 넣는다.
ㄴ Combo Box를 확인하고, 규칙을 정의한다
	ㄴ 선택항목이면서 Default값이 없을 경우 '전체' 혹은 '선택'의 기본 문구로 통일한다
	ㄴ 콤보 박스 버튼의 초기 설정값은 자주 사용하는 값을 우선으로 한다
	ㄴ 콤보 박스 버턴을 클릭하면 별도의 저장 버튼을 눌러서 저장되도록 한다.
ㄴ 라디오 버튼을 사용할 경우 텍스트와의 정렬이 맞도록 CSS를 적용한다
ㄴ 라디오 버튼의 초기 설절값은 반드시 자주 사용하는 값으로 우선한다
ㄴ 라디오 버튼을 클릭하면 자동으로 선택된 값이 저장되도록 한다.
