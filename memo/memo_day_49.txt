2. 물리데이터 저장소 설계하기
반정규화
ㄴ 조회 성능을 높이기 위해 데이터를 중복하거나 테이블 구조를 바꾸는 설계 방법
엔티티 -> 테이블
속성 -> 컬럼
UID ->기본키로 변환
관계 -> 외래키
컬럼의 타입과 길이를 정의
반정규화 적용 여부 검토



3. 물리 데이터 저장소  구성
ㄴ 설계에 따라 데이터저장소에 실제 데이터가 저장될 물리적 공간 구성
ㄴ 테이블간 관계가 설정되었을 떄 데이터가 잘못 저장되거나, 관계가 깨지는 것을 막기 위한 규칙

기본키 테이블의 데이터가 삭제될 때
ㄴ CASCADE : 기본키의 데이터가 삭제되면 참조하는 데이터도 삭제
ㄴ RESTRICT : 데이터가 참조되고 있으면 기본키의 데이터는 삭제할 수 없다
ㄴ NULLIFY : 기본키의 데이터가 삭제되면 참조키 데이터를 NULL로 변경

사용법
CONSTRAINT 외래키명 FOREIGN KEY (컬럼명)
REFERNCES 테이블명(컬럼명)
ON UPDATE 제약조건

인덱스 설계
적용되는 컬럼의 선정
분포도가 10 ~ 15%인 컬럼에 사용하는 것이 일반적임
분포도 = (1/컬럼의 종류) x 100

분포도가 범위 이상이라도 부분 처리를 목적으로 하는 경우
조회 및 출력 조건으로 사용되는 컬럼인 경우
기본키(PK)와 유일키(UNIQUE)는 인덱스가 자동으로 생성된다

분포도가 좋은 컬럼은 단독 인덱스로 생성한다
자주 조합되어 사용되는 컬럼은 결합 인덱스로 사용

수정이 빈번하지 않은 컬럼으로 선정

뷰
뷰 속성
REPLACE( CREATE OR REPLACE 뷰이름)
ㄴ 뷰가 이미 만들어져 있다면 기존 뷰를 새로운 내용으로 다시 생성하는 속성

FORCE
ㄴ 기본테이블이 없어도 뷰를 생성하는 속성
ㄴ 대신 뷰를 사용하려고 하면 에러 발생

NOFORCE
ㄴ 기본테이블이 없으면 뷰를 만들지 마라

WITH CHECK OPTION
ㄴ 뷰의 조건을 만족하는 행만 INSERT 또는 UPDATE할 수 있다

WITH READ ONLY
ㄴ 뷰를 조회 전용으로 만드는 속성

뷰 설계시 고려사항
ㄴ 복잡한 뷰는 성능이 느려질 수 있다
ㄴ 뷰 안의 SELECT 조건도 인덱스를 고려해야 한다

클러스터 설계
ㄴ 관련된 데이터를 물리적으로 가까운 위치에 저장하여 조회 성능을 높이는 기법

인덱스 : 목차
ㄴ 특정 행의 검색이 빠르다
클러스터 : 물리적으로 모아둠
ㄴ 범위 조회, 반복 조인 조회가 빠르다

클러스터 만드는 법
1. 클러스터 생성
CREATE CLUSTER 클러스터이름(
	컬럼명 타입;	
)
2. 클러스터 인덱스 생성
CREATE INDEX 인덱스명
ON CLUSTER 클러스터명;
3. 클러스터를 사용하는 테이블 만들기
CREATE TABLE 테이블명 (
	...
)
CLUSTER 클러스터이름(컬럼)

MYSQL은 오라클처럼 직접 만들지는 않는다.
PK를 기준으로 자동으로 클러스터형 인덱스 만들어진다

적용기준
ㄴ 분포도가 넓을수록(수치가 낮은건) 유리한 기법
ㄴ 클러스터는 여러 블록이상의 테이블에 적용한다
ㄴ 대량의 범위를 자주 조회할 떄 적용
ㄴ 여러 테이블이 빈번히 조회될 때

주의사항
ㄴ 검색은 빨라지지만 입력, 수정, 삭제는 느려질 수 있다

파티션
ㄴ 하나의 큰 테이블을 내부적으로 여러 조각으로 나눠 저장하는 기술
파티션 종류
ㄴ 범위 분할
	ㄴ 사용자가 데이터를 구간별로 나눈다
	ㄴ 데이터가 각 파티션별로 골구로 들어가있지 않을 수도 있다
	ㄴ 조회 범위를 줄일 수 있다
	ㄴ 오래된 데이터를 관리하기 쉽다
ㄴ 해시 분할
	ㄴ DB가 게산해서 각 파티션에 골고루 분배
	ㄴ 해시값 % 파티션 개수 -> 어떤 파티션에 넣을지 결정
	ㄴ 회원, 사용자, 고객 데이터 -> ID기반의 조회가 많은 시스템
ㄴ 조합 분할
	ㄴ 파티션 기준을 2개 이상 조합해서 데이터를 나누는 방식
	ㄴ 먼저큰 기준으로 한 번 나누고 그 안에서 다시 작은 기준으로 한 번 더 나누는 방식
	ㄴ 1차 파티션만으로 부족할 때
	ㄴ 날짜 조건과 고객 조건을 함께 자주 사용할 때
	ㄴ 특정 연도나 월에 데이터가 몰리는 문제를 줄일 수 있다

파티셔닝 순서
ㄴ 파티션 종류 결정(범위, 해쉬, 조합)
ㄴ 파티션 키 선정(데이터를 나눌 기준이 되는 컬럼)
ㄴ 파티션 수 선정

디스크 구성 설계
ㄴ 데이터를 어디에, 어떻게 저장할 것인지 계획하는 작업
ㄴ 정확한 용량을 산정하여 디스크 사용의 효율을 높인다
ㄴ 업무량이 집중되어있는 디스크를 분리하여 설계함으로써 
    집중화된 디스크에 대한 입출력 부하를 분산한다
ㄴ 입출력 경합을 최소화 하여 데이터의 접근 서능을 향상시킨다

물리 데이터 모델 구성하기

다양한 오브젝트를 설계(테이블, 제약조건, 인덱스, 뷰, 클러스터, 파티션, 테이블스페이스)
ㄴ 제약조건 설계 : 데이터 무결성 보장
ㄴ 인덱스 설계 : 검색 속도 향상
ㄴ 뷰 설계 : 재사용성과 보안
ㄴ 클러스터 설계 : 물리적 저장 최적화
ㄴ 파티션 설계 : 대용량 데이터 관리
ㄴ 디스크 구성 설계 : I/O 성능 최적화
