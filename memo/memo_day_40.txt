2. 개발 기술 환경  정의



BSD 
- 업적으로 상용할 수 있고, 수정해서 판매할 수도 있으며, 소스코드를 공개하지 않아도 된다
- 대신 저작권 표시와 라이선스 문구는 유지해야 한다

고려할 사항
가용성
- 장시간 시스템을 운영할 때 장애 발생 가능성
- DBMS의 버그 등으로 인한 패치 설치를 위한 재기동
- 백업 및 복구 편의성
- DBMS 이중화 및 복제 지원
성능
- 대규무 데이터 처리 성능
- 대량 트랜잭션 처리 성능
- 다양한 튜닝 옵션 지원
- 비용 기반 최적화 지원 및 설정의 최소화
기술지원
- 공급 벤더사드르이 안정적인 기술 지원
- 다수의 사용자들 간의 정보 공유
- 오픈소스 여부
상호호환성
- 설치 가능한 운영체제 종류
-다양한 운영체제에서 지원하는 JDBC
구축 비용
- 라이선스 비용
- 유지 관리 비용
- 총 소유 비용(TCO)

WAS(Web Application Server)
- 사용자의 요청을 받아 실제 프로그램을 실행하고 결과를 만들어주는 미들웨어

사용자가 사이트 접근한다
1. 변하지 않는 파일에 대한 요청(정적 데이터)
- 이미지
- 스타일(폰트, 색상, 디자인)
- 페이지의 구조(html)
이미 만들어져 있기 때문에(사용자에게) 전달만 하면 된다.
계산을 할 필요가 없다

2. 계산이 필요한 요청(동적 데이터)
로그인, 상품 검색, 게시글 작성, 주문하기 -> DB를 한 번씩 거쳐야 함

아이디 : test
비밀번호 : 1234
	1. 데이터베이스 확인
	2. 아이디 존재여부를 검사
	3. 비밀번호 일치 검사
	4. 로그인 성공 여부 결정
	5. 결과를 반환

이런 작업들은 웹 서버만으로는 어렵다

WAS 선택시 고려사항
가용성
- 시스템이 중단되지 않고 계속 사용할 수 있는가?
- 안정적인 트랜잭션 처리가 가능한가?
- WAS의 버그 등으로 인한 패치 설치를 위한 재기동이 적어야 한다
- WAS 이중화 지원
성능
- 대규모 트랜잭션 요청 처리를 할 수 있는가
- 다양한 설정 옵션을 지원
-GC(Garbage Collection) 옵션
기술지원
- 공급 벤더에서 안적적으로 기술 지원을 제공하는지
- WAS 사용자들이 얼마나 많고, 정보를 공유하는 지
구축비용
- 라이선스 비용(상용 WAS는 기능과 기술지원을 제공하는 대신 비용을 받는다.)
- 유지관리 비용(서버 운용 비용, 유지보수 인력 비용, 보안 패치 비용, 장애 대응 비용,
백업 비용)
- TCO(처음부터 끝까지 총 소유 비용) 초기 구매 비용 + 유지 보수 비용



오픈소스
- 소스코드가 공개되어있고, 자유롭게 사용가능한 프로그램
- 아무 제한 없이 사용이 되는 것은 아니고 라이선스 규칙을 따라야 한다
장점
비용 절감 : 무료로 사용 가능
자유로운 수정 : 기능 추가 가능
커뮤니티 활성화 : 자료가 풍부함
빠른 업데이트 : 개발자의 참여가 많다

단점
공식 지원 부족 : 제조사 지원 약할 수 있음
설정 난이도 : 직접 설정 필요
라이선스 확인 필요 : GPL 등 확인 필요


요구사항 확인
- 요구사항이 올바른지 점검하는 과정이다.

확인과 검증
Validation(확인)
- 만드려는 시스템이 고객이 실제 원하는 것인지 확인하는 것
Varification(검증)
- 작성된 요구사항 문서가 올바르게 작성되었는지 검사하는 것
- 고객이 원하는 방향이 맞는지도 중요하지만, 그 요구사항이 문서로 정확하고 일관성있게
정리되어 있어야 한다.

요구사항 문서 검증 시 확인 항목
1. 이해 가능성 : 누구나 이해 가능한가?
2. 일관성 : 내용끼리 서로 충돌하지 않는가?
3. 완전성 : 필요한 내용이 빠지지 않았는가?
4. 표준 준수 : 회사 규칙을 따르는가?


출결 등록/조회/수정 기능을 포함했다 -> ok
모바일을 원했는데 PC화면만 설계됨 -> 실패
관리자 승인 기능을 원했는데 자동 승인으로 설계됨 -> 실패

- 문서를 찾기 어렵다
- 유지보수가 어려워진다
- 협업 효율이 감소한다

요구사항 관리 툴
요구사항 문서를 엑셀이나 한글 파일로만 관리하지 않고,
변경 이력, 승인 상태, 담담자, 추적 관계까지 체계적으로 관리하는 도구

요구사항 관리툴을 사용하는 이유
- 요구사항은 프로젝트 중간에 계속 바뀐다
누가 바꿨는가?
언제 바꿨는가?
왜 바꿨는가?
승인되었는가?
어떤 기능에 영향이 있는가?
테스트도 바뀌어야 하는가?
위와 같은 내용을 관리해야 한다.

사람이 파일로 관리하면 빠트리기 쉽기 때문에 요구사항 관리툴을 사용한다

요구사항 관리툴이 하는 일

변경 이력 관리 : 누가, 언제, 무엇을 바꿨는지 기록
승인 관리 : 검토 중, 승인, 반려 상태 관리
추적성 관리 : 요구사항과 설계, 개발, 테스트 연결
담당자 관리 : 누가 처리할지 지정
우선순위 관리 : 중요도 설정
산출물 관리 : 요구사항 정의서, 명세서, 테스트 문서 관리

요구사항 확인 기법
작성된 요구사항이 올바른지, 빠진 것은 없는지, 
고객이 원하는 내용이 맞는지 확인하는 방법


1. 요구사항 검토
- 여러 사람이 모여 요구사항 문서를 읽고 문제를 찾는다.
오타, 잘못된 가정, 애매한 표현, 누락, 모순, 표준위반점을 찾는다.
- 검토자 그룹을 어떻게 구성하느냐가 중요하다.
예를 들어 고객 중심 프로젝트에서는 검토자 그룹에 고객 대표자가 1명 이상 포함되어야 한다.
- 검토는 시스템 정의서, 시스템 사양서, 소프트웨어 요구사항명세서를 
완성한 시점에서 이루어진다

시스템 정의서
- 무엇을 만드는지 큰 방향을 정의하는 문서(주요 기능만 정의)

시스템 사양서 
- 정의서보다 조금 더 구체적인 문서(그 기능들이 어떻게 동작할 것인지)

소프트웨어 요구사항 명세서
- 개발자가 실제 구현할 수 있을 정도로 상세하게 적은 문서

2. 프로토타이핑(Prototyping)
- 최종 제품을 만들기 전에 일부 기능이나 화면을 미리 만들어 
- 사용자가 확인하도록 하는 방법
- 처음에는  생각하지 못했던 새로운 요구사항을 도출할 수 있다.

장점 
- 분석가의 가정을 파악하고 잘못된 경우 유용한 피드백을 제공한다.
- 사용자 인터페이스의 동적인 행위가 문서나 그래픽 모델보다 프로토타입으로
- 이해하기 쉽다.
- 요구사항의 가변성이 프로토타이핑 이후에 급격히 감소한다.
- 잘못된 요구사항에 대해 자원 낭비 방지

단점 
- 핵심 기능보다 디자인에 집중하게 될 수 있다
- 프로토타입을 만드는 시간과 비용이 추가로 발생할 수 있다



프로토타이핑 종류
- 종이에 그름 : 가장 단순함

화면 프로토타입
- Figma와 같은 도구 사용

기능 프로토타입
- 일부 기능을 실제로 구현


모델 검증
- 개발 전에 만든 분석 모델이 올바르게 작성되었는지 확인하는 방법
- 개발 전에 모델 오류를 찾기 위해서 검증을 해야 한다
- 예를 들어 객체 모델의 경우 객체들 사이에 존재하는 의사소통 경를 검증하기 위해
- 정적 분석을 수행하느 ㄴ것이 유용하다

모델 : 실제 시스템 코드게 아니라 개발 전에 만드는 분석용 설계자료
모델의 종류
업무의 흐름도 : 사용자가 어떤 순서로 처리하는지 나타내는 그림
유스어케이스 다이어그램 : 사용자가 어떤 시스템으로 무엇을 할 수 있는지 나타낸그림
객체 모델 : 시스템 안에 어떤 속성들잉이 있고 서로 어떻게 연결되는지 나타낸 모델
클래스 다이어그램 :: 클래스, 속성, 메세더, 관계를 표현한 그림
데이터 모델 : 데이터베이스 어떤 데이터가 저장되는지 나타낸 모델


의사소통 경로
- 객체들끼리 정보를 주고받을 수 있는 연결 관계
예 : 도서대서 -> 도서 대여 시스템에서 회원이 도서를 빌리면 다음과 같이 작동한다
회원 -> 대여 -> 도서
회원이 대여를 요청하고 대여 객체는 도서 객체에서 이제 재미여부 확인해야한다
회원 		-> 대여	-> 도서
이렇게 되어있으면 대여 객체가 도서 정부를 확인할 길이 없다
필요한 의사소통 결로가 다 빠진 상태이다.


정적분석
- 프로그램을 직접 실행하는 것이 아닌, 구조만 보고 검사하는 방식
- 객체 모델을 실행할 수 없지만, 그리밍나 관계를 구조적으로 검사하는 것


인수 테스트
개발이 거의 끝난 후 만든 프로그램이 고객이 요구한 내용을 정말 만족했는지 확인하는 테스트
테스트
- 요구사항의 중요한 속성은 최종 제품이 요구사항을 만족시키는 기능인지 확인이 가능해야 한다
- 각각의 요구사항을 어떻게 확인할 것인지에 대한 계획을 세워야한다

회원은 로그인할 수 있어야 한다
테스트 계획
1. 아이디 입력
2. 비밀번호 입력
3. 로그인 버튼 클릭
4. 메인 페이지 이동 여부 확인

요구사항
회원은 상품을 주문할 수 있어야한다
테스트 계획
1. 상품 선택
2. 수량 입력
3. 주문 버튼 클릭
4. 주문 내역 생성 여부 확인
