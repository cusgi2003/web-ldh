UI 문제 


문제: 화면 설계 과정 없이 요구사항 분석 후 바로 시스템 개발(코딩)에 착수했을 때, 프로젝트 후반부에 발생할 수 있는 수정 비용 및 시간적 측면의 부작용을 초기 프로토타입 목적과 연결하여 서술하시오.

빠르게 제작이 가능해서 수정이 쉽기 때문이다.


문제: UI 구성 요소를 설계할 때, '저장' 버튼과 '취소' 버튼의 화면 내 배치 및 간격 규칙을 일관성(Consistency) 측면에서 어떻게 적용해야 하는지 그 설계 원칙을 서술하시오.

버튼의 크기가 너무 작거나, 디자인이 동일하면 저장버튼과 취소버튼을 처음 보는 사람 입장에선 구분하기 어렵기 때문이다.


문제: 본문 교재에서는 나쁜 오류 메시지의 예시로 'Error 500'을 제시했습니다. 사용성 측면에서 이를 지양해야 하는 이유를 좋은 오류 메시지가 갖춰야 할 3가지 요소(문제 상황, 해결 행동, 표현 방식)와 비교하여 서술하시오.

문제상황 : 무엇이 잘못됐는지 이해할 수 없다.
해결 행동 : 무엇을 해결해야 다음 단계로 넘어갈 수 있는지 표시
표현 방식 : 직관적인 일반 사용자의 언어를 사용한다.


문제: UI 구성 요소를 기능 기준으로 분류할 때, '아이디 입력창'이나 '비밀번호 입력창'과 같이 사용자로부터 데이터를 받아오는 역할을 하는 유형의 명칭을 쓰시오.
> 입력 컴포넌트

문제: 사용자가 화면을 조작했을 때 시스템이 특정 동작을 발생시키도록 만드는 '사용자 행동(예: 버튼 클릭, 스와이프 등)'을 일컫는 UI 흐름 설계 용어를 쓰시오.
> 이벤트

문제: 시각 디자인 요소를 최소화하고 구조, 버튼 위치, 메뉴 배치, 화면 이동 흐름 중심의 설계에 집중하기 위해 초기 프로토타입 단계에서 빠르게 작성하는 화면 설계도의 명칭을 쓰시오.
> 와이어 프레임

문제: 웹 브라우저 기반으로 구동되며, 별도 설치 없이 실시간 공동 작업이 가능하여 현재 UI/UX 모사형 프로토타입 제작에 가장 널리 사용되는 도구의 명칭을 쓰시오.
> 피그마
