데이터 모델링
- 현실 세계의 업무 내용을 데이터베이스에 저장할 수 있도록
데이터 구조를 설계하는 과정

모델링 필요한 이유
이상현상의 발생을 막아야 한다
1. 데이터 중복이 발생할 수 있다
2. 갱신 이상 -> 같은 정보가 여러개 저장되어 있을 때 전부 수정하지 않아서
데이터의 불일치가 발생하는 현상
3. 삭제 이상 -> 필요없는 데이터를 지우려다 필요한 데이터까지 삭제하는 경우
4. 삽입 이상 -> 필요한 데이터를 추가하려다 필요없는 데이터까지 추가하는 경우

데이터 모델링 과정
1. 요구사항 분석
- 업무에서 어떤 데이터를 관리해야 하는지 파악하는 단계
- 주요 명사를 찾아서 -> 엔티티로 만든다
2. 개념적 모델링
- 업무 내용을 큰 그림으로 표현하는 단계
- 주로 ERD를 사용한다

주요 구성 요소
	ㄴ엔티티(Entity) : 관리해야 할 요소
	ㄴ속성(Attribute) : 엔티티가 가진 정보
	ㄴ관계 : (relationship)) : 엔티티 간 연결
	ㄴ식별자 (Identifier) : 데이터를 구분하는 값

논리 모델링
	- 개념적 모델을 실제 테이블 구조에 가깝게 바꾸는 단계
	- ERD를 가지고 테이블을 설계
	- 테이블의 이름이나 속성의 이름을 정하는 단계
	- 관게를 문장으로 처리

관계 차수 정하기
	1:1, 1:N, M:N 중 어떤 관계인지 정한다


정규화

실무에서는 보통 3차까지 다룸 -> 더 하면 오히려 조회속도가 느려짐
제 1 정규형(1NF) -> 속성 하나에는 하나의 값만 들어가야 한다
제 2 정규형(2NF) -> 복합키에서 일부 컬럼에만 의존하는 컬럼을 제거(부분 함숨 종속)
제 3 정규형(3NF) -> 일반 컬럼이 다른 일반 컬럼을 결정하면 안된다
(이행 함수 종속 X -> Y, Y -> Z, Z -> X)

BCNF (Boyce- Codd Normal Form)
- 제 3 정규형을 좀 더 엄격하게 적용한 정규형
- 모든 함수적 종속에서 결정자는 반드시 후보키여야 한다
	※ 결정자 : 관계형 데이터베이스에서 다른 속성의 값을 유일하게 결정짓는 
	속성들의 집합을 의미한다 ( A -> B)
	※ 후보키 : PK가 될 수 있는 속성

제 3 정규형을 지켜도 이상 현상이 남아있을 수 있다

제 4 정규형(4NF) -> 다중값  종속 제거  
한 엔티티가 서로 독립적인 여러 값 집합을 가지면 분리

제 5 정규형(5NF) -> 조인으로 복원 가능한 경우 분리


물리모델링
- 논리적 모델을 실제 DBMS에서 실행 가능한 SQL로 바꾸는 것
	1. 컬럼 타입 구체적으로 정하기
	2. NULL 허용 여부 정하기
	3. 중복 불가 조건 정하기
	4. 기본 값 정하기
	5. 외래키 제약조건 정하기


DML(Data Manipulation Language) : 데이터 조작어
- 데이터에 대해서 조회, 수정, 추가, 삭제하는 명령어
	NULL값이 들어있는 데이터 조회
		- IS NULL
	NULL값이 없는 데이터 조회
		- IS NOT NULL

like : 문자열 안에서 특정 글자나 패턴을 찾을 때 사용하는 조건문
select 컬럼명 from 테이블명 where 컬럼명 like "패턴"
ㄴ 포함 여부, 시작 문자, 끝 문자, 특정 글자 패턴을 찾을 수 있다

% : 글자 수와 상관없이 아무 문자나 여러개 올 수 있다는 뜻

'A%' : A로 시작하는 데이터
'%A' : A로 끝나는 데이터
'%A%' : A를 포함하고 있는 데이터
'A%B' : A로 시작하고 B로 끝나는 값

_ : 문자 1개만 의미
'A_' : A로 시작하고 전체 길이가 2인 값
'A__' : A로 시작하고 전체 길이가 3인 값
'_A' : A로 끝나는 전체 길이가 2인 값
'_A_' : 전체 길이가 3이고, A가 가운데 있는 값


DCL(Data Control language) : 데이터 제어어
- 데이터베이스에서 사용자 권한을 관리하는 SQL
GRANT : 권한을 부여
REVOKE : 권한 회수

DCL이 필요한 이유
- 모든 사용자에게 모든 권한을 주면 위험해진다

MYSQL에서는 사용자를 다음과 같이 구분한다
'사용자명'@'접속자위치'
'teacher'@'localhost'

사용자 생성하는 법
create user '사용자명'@'접속자위치'
identified by '비밀번호';
