UI(UserInterface)
ㄴ 제품이나 소프트웨어와 상호작용할 때 직접 보거나 조작하는 화면 요소
ㄴ 사용자가 눈으로 보거나 손으로 누르는 모든 구성요소

UX(UserExperience)
ㄴ 사용자가 소프트웨어를 사용하면서 느끼는 전체적인 경험
ㄴ 서비스를 이용하면서 느끼는 편리함, 만족감, 효율성, 즐거움 등을 모두 포함한다

---------------------------------------------------------------------------------------------------

UI 구성요소
1. 시각적 디자인(Visual Design)
	ㄴ 사용자가 화면에서 보는 모든 요소
	ㄴ 색상, 글꼴, 아이콘, 이미지, 레이아웃
2. 상호작용 디자인(Interaction Design)
	ㄴ 사용자의 행동에 대해 시스템이 어떻게 반응하는지 설계
	ㄴ 좋은 상호작용 디자인은 사용자가 어떤 상태인지 쉽게 알 수 있도록 도와준다
3. 정보구조(Information Architecture)
	ㄴ 정보를 체계적으로 분류하고 배치하는 방법
	ㄴ 사용자가 원하는 정보를 쉽게 찾을 수 있도록 구성하는 것

UX 구성 요소
1. 사용성(Usability)
	ㄴ 사용자가 얼마나 서비스를 쉽게 이용할 수 있는가
2. 유용성(Usefulness)
	ㄴ 서비스가 사용자의 요구사항을 얼마나 잘 충족하는지 의미
3. 매력도(Desirability)
	ㄴ 서비스가 사용자에게 얼마나 매력적으로 느껴지는가
4. 접근성(Accessibility)
	ㄴ 모든 사용자가 서비스를 이용할 수 있도록 만드는 정도
	ㄴ 시각장애인, 청각장애인, 고령자, 색약 사용자 등도 서비스를 사용할 수 있어야 한다
5. 신뢰성 (Credibility)
	ㄴ 사용자가 서비스와 정볼르 얼마나 믿을 수 있는가

UI/UX의 관계
	ㄴ 좋은 UI는 좋은 UX에 도움이 된다.
	ㄴ 항상 좋은 UI가 좋은 UX는 아니다.

UI/UX 표준
ISO 9241-210
	ISO : 국제 표준화 기구
	9241 : 사람과 컴퓨터 상호작용 관련 표준
	210 : 사용자 중심 설계 분야

	적용대상 : 사용자와 컴퓨터가 상호작용하는 모든 분야
		ㄴ 소프트웨어(프로그램)
		ㄴ 웹사이트
		ㄴ 모바일 앱
		ㄴ 디지털 기기

주요내용(사용자 중심 설계를 위한 6가지 원칙)
1. 사용자는 설계 과정 전반에 적극적으로 참여한다
	ㄴ 실계 사용자의 의견을 먼저 수집한다
2. 사용자 중심 설계는 전체 사용자 경험을 고려해야 한다
	ㄴ 기능만 동작한다고 되는게 아니라 사용자가 느끼는 감정도 고려
3. 설계는 반복적으로 이루어져야 한다
	ㄴ 계속 테스트를 하고 수정해서 UI를 만들어나가야 한다
4. 설계는 사용자와 소프트웨어 구축 목표 및 환경을 기반으로 한다
	ㄴ 누가, 어떤 환경에서 사용하는지 고려해야 한다
5. 사용자 중심 설계 팀은 여러 학문 분야가 함께 협력해야 한다.
	ㄴ UI/UX는 한 사람이 만들 수 없다
	ㄴ 여러 분야의 사람이 함께 참여해야 한다
		기획자 : 요구사항 분석
		UI 디자이너 : 화면 디자인
		UX 디자이너 : 사용자 경험 설계
		개발자 : 기능 구현
		관리자 : 업무 요구사항 제공
6. 사용자 평가에 반드시 포함되어야 한다
	ㄴ 실제 사용자가 평가를 하고 개선해야 한다.

UI 요구사항 확인하기
	ㄴ 화면을 설계하기 전에 "사용자가 어떤 화면을 원하며, 어떤 방식으로 사용할 것인가"를 분석하는 과정
	ㄴ 소프트웨어를 개바랗기 전에 화면에 대한 요구사항을 먼저 파악해야 한다

요구사항 확인 없이 개발을 시작한다면 다음과 같은 문제가 발생할 수 있다.
	개발자 : 로그인 기능만 있으면 되겠지 ?
	사용자 : 회원가입, 아이디 찾기, 비밀번호 찾기, 소셜로그인 기능이 필요
		

1. 소프트웨어 개발을 위한 사용자 화면 표준 및 지침 확인

1-1. 국가기관에서 제공하는 UI/UX 가이드라인 수집 분석
	ㄴ 국가 기관의 공식 웹사이트에 접속
	ㄴ UI/UX 가이드라인 확인
	ㄴ 최신 버전 문서를 우선 확인

1-2. 제공하는 지침의 핵심 항목을 발췌하여 정리한다
	ㄴ 자료의 주요 항목 선별 기준을 정의
	접근성 : 장애인, 고령자 등 다양한 사용자가 서비를 이용할 수 있도록
	일관성 : 비슷한 기능은 동일한 방식으로 제공해라
	색상과 대비 : 글자와 배경이 얼마나 잘 구분되는가

1-3. 공공기관 웹사이트/앱 설계에 적용된 가이드라인 요소 분석
	ㄴ 분석대상 선정 (정부24, 홈택스, 국민신문고, 워크넷, ...)
	ㄴ 분석 항목을 정리
	ㄴ 분석 기준 설정
		색상 대비 4.5 : 1 이상
		이미지 대체 텍스트 제공
		키보드로 이동 가능
	ㄴ 직접 사용해보고 준수 여부를 확인한다.
	ㄴ 준수 사례와 미흡 사례를 분류한다
	ㄴ 문제점과 개선 방안을 도출한다.

1-4. 사용자 화면의 네비게이션 구성, 콘텐츠 배치 등에 대한 지침을 요약 정리한다
	ㄴ 내비게이션 : 사용자가 화면을 이동하는 방법(메뉴)	
		ㄴ 메뉴 위치가 일정하다
		ㄴ 이동 경로가 단순하다
	ㄴ 콘텐츠 배치 : 화면에 정보를 어떻게 배치할 지에 대한 설계 방식
		ㄴ 상단 : 제목
		ㄴ 중앙 : 주요 내용
		ㄴ 하단 : 추가 정보
	ㄴ 내비게이션 구성 및 콘텐츠 배치 원칙 분석 결과를 정리하여 문서화
	ㄴ 실사용자, UI 기획자, UI 디자이너와 검토하여 의견을 수렴한 후 최종 지침서 확정하기


2. 국제 UI/UX 표준 규격을 수집하고 분석한다

2-1. 주요 국제 표준화 기구의 UI/UX 관련 규격을 조사하여 목록화 한다.
	UI/UX와 관련된 주요 기관
		ISO : 국제 표준화 기구, 다양한 산업 분야의 국제 표준을 만든다
		W3C : 웹 기술 표준을 만드는 기관
		WCAG : 웹 접근성 지침으로, 
		IEEE : 전기, 저낮, 소프트웨어 분야의 표준을 만드는 기관
	ㄴ 주요 국제 UI/UX 표준 규격 목록
		ISO 9241-210 : 사용자 중심 화면 설계, UX 원칙 포함
		WCAG 2.2 : 웹 콘텐츠 접근성, A/AA/AAA 등급 기준
		IEEE 830 : 소프트웨어 오규사항 명세, UI 요구사항 명세 관련 내용

2-2. 주요 표준별 핵심 내용을 요약한다
	ㄴ 각 표준이 무엇을 위한 것인지, 어디에 적용되는지, 어떤 내용을 포함하는지 요약해야 한다

2-3. 국내 적용 사례를 수집하고 비교 분석한다
	ㄴ 국제 표준은 넓은 기준을 제공
	ㄴ 국내 프로젝트에서는 국내법, 공공기관 지침, 웹 접근성 기준을 함께 고려해야 한다

2-4. 표준별 적용 가능성 및 제약사항을 도출하여 정리한다
	ㄴ 표준을 조사한 뒤, 실제 프로젝트에 적용할 수 있는지 판단해야 한다
	ㄴ 모든 표준을 그대로 적용하기 어렵다
	ㄴ 프로젝트의 예산, 일정, 인력, 기술 수준에 따라 적용 범위가 달라질 수 있다


3. 플랫폿별 디자인 가디으라인을 수집하고 분석한다

3-1 . 분석 대상 주요 플랫폿음 선정하고특성 파악하기
	플랫폿 : 소프트웨어가 실행되는 환경
	ㄴ 플랫폼별 특정 쉽게 이해하기
		모바일 : 버튼이 너무 작으면 사용하기 힘듦
		웹 : 접속한 화면에 따라 레이아웃이 적절하게 바뀌어야 한다
		데스크톱 : 키보드와 마우스 중심으로 조작
		스마트워치 : 빠른 조작 중슴으로 설계

3-2. 각 플랫폼별 공식 디자인 가이드라인 문서를확인한다
	ㄴ 플랫폿마다 사용자가 익속하게 느끼는 화면 방식이 다르기 때문
	ㄴ 가이드라인에서 확인할 내용
	ㄴ 색상 : 서비스의 분위기와 정보 전달에 영향을 준다
	ㄴ 타임포그래피 : 글꼴, 글자, 크기, 줄 간격
	ㄴ 레이아웃 : 화면 배치 방식
	ㄴ 컴포넌트 : 버튼, 입력창, 카드, 메뉴처럼 사용하는 UI 기능
	ㄴ 상호작용 : 사용자가 버튼을 누르거나 스와이프할 때 시스템이 어떻게 반응할 지 결정

3-3. 플랫폼별 공통 요소와 차이점을 비교 분석한다.
	ㄴ 같은 기능이라도 플랫폼에 따라 표현 방식이 달라질 수 있다.

3-4. 플랫폼별 적용 사례를 조사하고 분석
	ㄴ 가이드라인을 읽는 것만으로는 실제 화면을 설계하기 어렵다
	ㄴ 실제 서비스에서 가이드라인이 어떻게 반영되었는지 확인하면 설계 감각을 키울 수 있다.
	ㄴ 사례 분석 항목
		화면 구조 : 정보가 어떻게 배치되어 있는가?
		메뉴 구성 : 사용자가 쉽게 이동할 수 있는가?
		버튼 스타일 ; 버튼이 명확하고 누르기 쉬운가?
		입력 방식 : 터치, 키보드, 마우스, 리모컨에 적합한가?
		피드백 : 클리깅나 선택 후 반응이 명확한가?
		접근성 : 글자 크기, 색상 대비, 보조 기능을 고려했는가


4. 기존 유사 응용 소프트웨어의 UI 설계 사례를 조사하고분석한다
	ㄴ 새로운 서비스를 설계할 때 모든 것을 처음부터 만드는 것은 매우 어렵다.

4-1. 유사 응용소프트웨어를 선정하고 분석한다
	ㄴ 비슷한 서비스를 분석하면 어떤 메뉴 구조를 사용하는지, 어떤 화면 구성을 사용하는지,
	    사용자가 어떤 기능을 자주 사용하는지 파악할 수 있다.

4-2. 각 소프트웨어의 주요 UI 구성 요소를 분석한다
	ㄴ 분석해야 하는 항목
		레이아웃 : 화면 배치 방식
		내비게이션 : 메뉴 이동 구조
		버튼 스타일 : 버튼 디자인
		피드백 요소 : 메시지, 알림, 상태 표시

	사이드바 사례를 분석하고 특성을 정리한다
	ㄴ 메뉴 이동이 빠르다
	ㄴ 많은 기능을 정리할 수 있다
	ㄴ 작은 화면에서는 공간을 많이 차지한다
	ㄴ 모바일 환경에서는사용하기 어렵다

	다크모드 사례를 분석하고 특성을 정리한다
	ㄴ 눈에 피로가 감소
	ㄴ 베터리 절약
	ㄴ 어두운 환경에서 사용 편리
	ㄴ 색상 설계가 어려움
	ㄴ 가독성이 떨어질 수 있다.

	카드 레이아웃을 분석하고 특성을 정리한다
	ㄴ 정보 구분이 쉽다
	ㄴ 모바일 환경에 적합하다
	ㄴ 확장성이 좋다
	ㄴ 많은 정보를 한 번에 보기 어렵다
	ㄴ 공간을 많이 사용한다

	모달(Modal) 사례를 분석하고 특성을 정리한다
	ㄴ 사용자가 집중을 유도한다
	ㄴ 중요한 정보를 강조할 수 있다
	ㄴ 너무 많이 사용하면 불편하다
	ㄴ 작은 화면에서는 공간이 부족하다

4-3. 사용자 편의성과 UX 요소 비교 평가를 한다
	ㄴ 단순히 UI를 분석하는 것만으로는 부족하다
	ㄴ 사용자 입장에서 얼마나 편리하지 평가해야 한다.

	평가 항목
		사용성 : 쉽게 사용할 수 있는가
		유용성 : 필요한 기능을 제공하는가
		접근성 : 누구나 사용할 수 있는가
		일관성 : 같은 규칙을 유지하는가
		만족도 : 사용자가 만족하는가

4-4. 사례 분석을 통해 UI 설게 트렌드와 패턴을 도출한다
	트렌드 : 현재 많은 서비스가 공통적으로 사용하는 설계 방식
	패턴 : 반복적으로 사용되는 UI 설계 방법
		ㄴ 최근 트렌드
			다크모드 지원
			카드 레이아웃 사용
			모바일 우선 설계
			간결한 메뉴 구조
			직관적인 아이콘 사용


5. 목표 응용소프트웨어의 UI요구사항을 확인한다

5-1. 목표 응용 소프트웨어의 기능 및 사용자를 분석한다
	ㄴ 소프트웨어 구축 목적 및 주요 기능 문서를 검토한다.
	검토 대상 문서
		기획서 : 프로젝트 목적
		제안서 : 구축 필요성
		회의록 : 요구사항 논의 내용
		요구사항 명세서 : 구현할 기능 목록
		업무 프로세스 문서 : 현재 업무 흐름

5-2. 목표 응용 소프트웨어의 기능 목록을 작성하고 이를 유형으로 분류
	ㄴ UI는 기능을 제공하기 위한 수단
	ㄴ 따라서 어떤 기능이 있는지 먼저 정리해야 한다.
	
	분류하는 이유
	ㄴ 기능 유형에 따라 화면 설계 방식이 달라지기 때문이다.

5-3. 목표 응용 소프트웨어의 사용자를 식별하여 정의한다
	ㄴ 같은 기능이라도 사용자에 따라 필요한 화면이 다르다
	ㄴ 사용자 특성에 따라 UI 복잡도가 달라진다
		IT 활용 능력이 낮음 -> 단순한 화면 필요
		IT 활용 능력이 높음 -> 다양한 기능 제공이 가능

5-4. 사용자 유형별 사용 목적 및 기대 행동을 정의한다.
	ㄴ 목적을 이해하면 필요한 메뉴와 화면을 결정할 수 있다

5-5. 사용자 기기, 운영체제, 이용 시간대 등 사용자 환경을 분석한다
	사용자 환경 : 사용자가 서비스를 사용하는 환경
	분석 항목
	기기(Device) : PC, 노트북, 태블릿, 스마트폰
	운영체제(OS) : Windows, macOS, Android, IOS
	이용장소 : 사무실, 학교, 집, 야외
	이용시간대 : 업무 시간, 야간, 주말

5-6. 사용자 경험상의 주요 고려사항을 도출한다
	ㄴ 사용자가 서비스를 사용할 때 느끼는 편의성과 만족도를 고려

	클릭수 최소화 : 목표까지 도달하기 위해, 많은 클릭은 피곤하다.
		ㄴ 명확환 피드백 제공 : 클릭해도 반응이 없으면 저장 여부를 알 수 없다.
	색상 대비 확보 : 배경과 글씨가 구분이 잘 되도록 한다.
	일관성 유지
	접근성 고려 : 대체 텍스트, 키보드 사용, 색약 모드 지원

5-7. 기능별 사용자 사용 시나리오 작성하기
	ㄴ 사용자가 어떤 순서로 화면을 사용하게 되는가
	ㄴ 사용 시나리오를 작성하면 어떤 화면이 필요한지, 버튼이 필요한지, 어떤 순서로 이동하는지 쉽게 파악할 수 있다.


6. 업무의 특성을 고려하여 UI 구성 요소 및 흐름 파악하기

6-1. 화면별 주요 구성 요소인 버튼, 폼, 메뉴등의 목록 작성하기
	ㄴ 각 구성 요소의 명칭, 기능, 위치 등을 정리하여 응용 소프트웨어의 각 화면에
	    포함된 UI 구성 요소를 식별하고 목록화 한다

6-2. 화면간 이동경로인 UI 흐름도를 작성한다
	ㄴ 사용자가 화면과 화면 사이르 어떻게 이동하는 표현한 그림
	ㄴ 사용자 이동 경로 : 사용자가 원하는 기능까지 몇 번 클릭하는지
	ㄴ 화면 간 관계 : 현재 화면 -> 다음 화면
	ㄴ  예외사항 : 로그인 실패 -> 로그인 화면 유지
	
	UI 흐름도를 작성하는 이유
	ㄴ 화면 누락 방지
	ㄴ 기능 누락 방지
	ㄴ 개발 범위 파악
	ㄴ 사용자 경험 개선

6-3. 핵심 기능인 상호작용 방식을 분석한다
	ㄴ 상호작용 : 사용자가 소프트웨어를 조작했을 때 시스템이 반응하는 과정
	ㄴ 같은 기능이라도 상호작용 방식에 따라 사용자의 편의성이 크게 달라진다

6-4. 디자인 일관성, 접근성, 사용성을 평가한다
	ㄴ 전체 UI에서 색상, 폰트, 버튼 형태 간격 등이 일관성이 있게 적용되었는지 확인하고 장애인이나
	    고령자 등도 쉽게 사용할 수 있는지 확인

6-5. 경쟁 소프트웨어의 UI 구성을 비교 분석
	ㄴ 경쟁 서비스의 장점을 참고하고 단점을 개선하기 위해서 

	분석 항목
	내비게이션 메뉴 : 사용자가 이동하는 방식
	버튼 디자인 : 버튼 크기, 색상, 형태
	입력 폼 : 사용자가 데이터를 입력하는 방식
	사용자 흐름 : 작업 수행 과정


7. 사용자 면담 및 UI/UX 요구사항 수집하기
	ㄴ 개발자나 디자이너가 사용하는 좋은 화면과 실제 사용자가 원하는 화면은 다를 수 있다.

7-1. 사용자 면담 또는 서룸ㄴ 대상자를 선정
	ㄴ 시스템 구축 목표와 관련된 주요 사앙자 그룹을 정의
	ㄴ 사용자 그룹별 특성과 사용 목적을 파악
	ㄴ 설문 대상자의 표본 크기와 선정 기준을 확정
	ㄴ 설문 참여자 모집 및 일정 조율(이메일, 문자, 전화, 사내 공지)

7-2. 면담 질문지 또는 설문지를 작성하고 계획을 수립한다
	ㄴ 면담/설문 목적과 목표를 명확하게 정의한다.
	목적예시 : 현재 시스템의 불편사항 파악, UI 개선 요구사항, 사용자 만족도
		목표 예시 : 사용자 만족도 80% 이상 달성	
	ㄴ 질문의 초안을 작성한다.
	기본 정보 : 연령대, 사용기기, 사용 경험
		현재 사용 경험 : 현재 UI 만족도, 자주 사용하는 기능, 불편한 화면
		전반적인 개선 사항 : 가장 먼저 개선해야할 사랑
	ㄴ 질문의 유형을 정의한다
		객관식
		주관식
		혼합형 질문
	ㄴ 설문지/면담 질의서 템플릿을 작성한다.
	ㄴ 면담/설문 계획을 수립한다.

7-3. 면담/설문을 사전에 검증하여 수정 보완한다.
	ㄴ 내부 검토를 통해 질문의 내용의 적절성을 평가한다
		ㄴ 질문의 명확한지?
		ㄴ 중복 질문은 없는지?
		ㄴ 빠진 질문은 없는지?
	ㄴ 소수의 사용자에게 파일럿 면담/설문을 진행한다.
		ㄴ 파일럿 조사 : 실제 조사 전 소규모로 먼저 테스트 하는 것
	ㄴ 이해하기 어려운 질문, 중복 항목, 누락된 질문을 파악

7-4. 사용자 대상 면담 및 설문을 진행한다.
	ㄴ 조사 목적과 일정을 공유한다.
	ㄴ 정의한 면담 방식에 따라 면담을 수행한다
	ㄴ 설문을 진행할 경우 온라인 링크 또는 오프라인으로 설문지를 배포한다.
	ㄴ 면담 및 설문 응답 결과를 취합
	ㄴ 면담 및 설문 참여자 보상과 결과에 대한 피드백을 공유한다

7-5. 기능 요구사항과 UI 개선 요구사항을 구분한다
	ㄴ 수집된 응답 내용을 기능/디자인으로 분류한다
	ㄴ 기능 개선 요청과 UI 개선 요청으로 태깅한다
	ㄴ 분류 기준과 예시를 작성한 후 데이터를 정리한다

7-6. 유사 서비스 등에서 발생한 반복적 불만/요구사항의 우선순위를 도출한다.
	ㄴ 경쟁사 및 유사 서비스의 의견을 조사한다
	ㄴ 반복적으로 나타나는 불만 사항을 정리한다
	ㄴ 발생 빈도와 영향도에 다른 우선 순위를 문서로 작성한다
	ㄴ 우선 순위에 따른 개선 아이디어를 도출하고 보고서를 작성한다.


8. UI 요구사항을 도출하고 정리한다

8-1. 수집된 요청사항과 유사 서비스 등에서 발생한 불만 사항등을 분석하여 결과를 통합한다.
	ㄴ 사용자 설문, 면담, 로그 데이터 등에서 수집된 요청 사항을 목록화한다.
	ㄴ 유사 서비스 리뷰 및 피드백 데이터를 수집한다.
	ㄴ 중복 유사 요청 사항을 그룹핑하여 항목을 정리한다
	ㄴ 분석 결과를 통합 보고서로 작성한다.

8-2. 사용 목적과 시나리오별 UI 요구사항을 도출한다
	ㄴ 사용자그룹별 사용 목적을 분석한다
	ㄴ 대표 사용 시나리오를 작성한다
	ㄴ 시나리오별 필요한 UI 구성 요소를 도출한다
	ㄴ 기능 흐름을 분석하여 UI 요구사항을 식별한다

8-3. 기능 요구사항과 비기능 요구사항을 분류한다
8-4. UI 요구사항에 고유 ID를 부여하고 설명을 작성한다
8-5. 요구사항의 우선순위와 난이도, 중요도 기준을 정의한다
8-6. UI 요구사항 명세서를 작성한다.
