기술적 위험 분석 검토

	문제가 생겼을 떄 영향이 얼만 큰지 판단하는 과정

	개발이 너무 어렵다
	조직 안에서 이 기술을 써 본 사람이 없다
	특정 업체 기술에만 의존
	라이선스 비용이 예상보다 크다
	기존의 아키텍처와 맞지 않다
	외부지원을 받을 수 없다

	복잡성

		하드웨어, 소프트웨어, 솔루션이 현재 시스템 아키텍처와 맞지 않는지도 확인해야 한다

	백엔드 : Spring boot
	프론트 : React
	클라우드 : AWS EC2

	특정 클라우드에서만 동작하는 전용 솔루션을 도입하려고 한다

분석 모델의 시스템화 타당성 분석
	분석모델이 실제 시스템으로 구현하는 것이 가능한지 판단

검토관점
필요한 자원
	시스템을 구현하고 운영하기 위해 얼마나 많은 컴퓨터 자원과 개발 자원이 필요한지 확인

	상호운용성
		우리 시스템이 다른 시스템과 정보를 주고받을 수 있는지 의미
	시장 성숙도 및 트렌드 부합성 검토
		사용하는 기술이 충분히 안정적이고 많이 사용되고 있는지 확인하는 것
		트렌드 부합성은 현재 개발 환경이나 산업 후름에 맞는 기술인지 확인하는 것
		기술을 잘못 선택하면 개발 중 문제가 생길 수 있다
	기술적 위험 분석
		개발 과정에서 문제가 될 수 있는 위험 요소를 미리 찾는 것이다

타당성 분석 수행
	순서
	1. 분석모델까지 요구사항 추적표를 작성하고, 타당성 검토 의견 속성을 추가한다.
	2. 작성된 요구사항 추적표에 타당성 검토 의견을 작성한다.
		ㄴ 성능 및 용량
		ㄴ 시스템 간 상호 운용성
		ㄴ 시장 성숙도 및 트렌드 부합성
		ㄴ 기술적 위험성 분석
		을 참조하여 검토 의견을 작성한다
	
	각 모델별로 유의해야 할 점
		유스케이스 모델 타당성 검토
			ㄴ 관련 액터들이 너무 많지는 않은지
			ㄴ 이벤트 흐름과 예외 흐름이 지나치게 복잡한지는 않은지
		개념적 수준 분석 클래스 모델 타당성 검토
			ㄴ 너무 많은 클래스 모델드이 도출되지 않았는지
		분석 클래스 모델 타당성 검토
			ㄴ 요구사항을 만족시키기 위한 경계/제어/엔티티 클래스들이 너무 많지는 않은지
			ㄴ 관계가 너무 복잡하지는 않은지
			ㄴ 관리해야할 속성들이 지나치게 많은지 검토

		※ 개념 수준 분석 클래스 모델
			ㄴ 요구사항을 읽고 시스템에서 관리해야 할 중요한 사람, 사물, 정보, 업무 개념을
			    클래스로 뽑은 것

Ex ) 도서 대여 시스템

※ 분석 클래스 모델
	요구사항을 만족시키기 위해 필요한 클래스들을 역할에 따라 정리한 모델	
경계 클래스 : 사용자나 외부 시스템과 만나는 클래스
제어 클래스 : 기능 흐름과 처리 순서를 담당하는 클래스
엔티티 클래스 : 실제 업무 데이터를 저장하고 관리하는 클래스

타당성 분석 결과를 이해관계자가 검증한다
1. 타당성 분석 결과를 관련 이해관계자에게 배포하여 사전 검토를 요청한다
2. 관련 이해관계자가 모여 분석모델 타당성 분석 결과를 검증한다
3. 타당성 분석 결과에 이견이 있는 경우 프로젝트 관리자의 중재하에 합의를 도출한다.

관련 이해관계자 검증을 거친 타당성 분석 결과를 확인하고 배포 및 공유한다.
1. 관련 이해관계자 검증을 거친 타당성 분석 결과를 의사 결정자가 확인한다
2. 의사 결정자가 확인한 타당성 분석 결과를 이해관계자에게 배포하여 공유한다
