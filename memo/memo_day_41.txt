- 잘못 해석을 하면 시간낭비, 비용증가로 이어질 수 있다.



확인과 검증

확인
- 고객의 목적과 요구사항이 맞는지 확인
검증
- 문서가 올바르게 작성되었는지 검사하는 것


검증시 확인하는 항목
이해 가능성
- 읽는 사람이 같은 의미로 이해할 수 있는가?
- 애매한 표현이 있으면 사람마다 다르게 해석할 수 있다.

사용자는 "빠르게" 로그인 할  수 있어야 한다 -> 애매한 표현
사용자는 로그인 버튼을 누른 후 3초 이내에 로그인 결과를 확인할 수 있어야한다 -> 구체적

요구사항
사용자는 회원가입할 수 있어야 한다

빠진 내용
입력항목이 무엇인지?
비밀번호 조건이 뭔지?
중복 확인?
실패 시 처리?

표준 준수
- 요구사항 문서가 회사나 프로젝트에서 정한 규칙을 지키는 것을 의미한다
- 표준을 지키지 않으면 문서를 작성하는 사람마다 형식이 달라져 이해하기 어렵다.


요구사항 확인 기법
- 작성된 요구사항이 올바른지, 빠진 것은 없는지, 고객이 원하는 내용이 맞는지 확인하는 기법

확인의 목적
오류 발견 - 잘못 작성된 요구사항 찾기
누락 확인 - 빠진 기능이나 조건이 없는지 확인
충돌 제거 - 서로 모순되는 요구사항을  찾는다.
이해 일치 - 고객, 분석가, 개발자가 같은 의미로 이해하는지 확인
테스트 가능성 확보 - 나중에 결과물을 검증할 수 있도록 요구사항을 구체화한다.

-----------------------------------------------------------------------------------------

요구사항 확인 기법의 종류
1. 요구사항 검토
- 작성된 요구사항 문서를 여러 사람이 함께 검토하여 오류, 잘못된 가정, 애매한 표현, 누락사항,
표준 위반 등을 찾아내는 활동
- 검토자 그룹의 구성 : 고객의 프로젝트라면 여러 그룹의 대표자가 참여하는 것이 좋다
	고객 대표 : 실제 원하는 기능이 맞는지 확인
	사용자 : 사용하기 편한지 확인
	분석가 : 요구사항 누락 여부 확인
	개발자 : 구현 가능한지 확인
	테스터 : 테스트 가능한지 확인
	프로젝트 관리자 : 일정/비용 영향 확인

개발자 끼리만 검토할 경우 개발자들끼리 원하는 기능만 만들 위험이 있다.


요구사항 검토는 시스템 정의서, 시스템 사양서, 소프트웨어 요구사항 명세서를 완성한 시점에
이루어진다.

2. 시스템 정의서 
- 무엇을 만드는지 큰 방향을 정의하는 문서
시스템 사양서
- 정의서보다 조금 더 구체적인 문서
요구사항 명세서
- 개발자가 실제 구현할 수 있을 정도로 상세하게 적은 문서

3. 프로토타이핑
- 최종 제품을 만들기 전에 일부 기능이나 화면을 미리 만들어서 사용자가 확인할 수 있도록 하는 방법
- 소프트웨어 요구사항에 대해 개발자가 해석한 것을 확인할 수 있다.
- 문서만으로 생각하지 못했던 새로운 요구사항을 도출할 수 있다.

장점
- 문서보다 이해하기 편하다
- 요구사항의 변경이 프로토타이핑 이후에 많이 감소한다.
- 잘못된 요구사항에 대해 자원 낭비를 방지할 수 있다.

단점
- 핵심 기능보다 디자인에 집중하게 될 수 있다.
- 프로토타입을 만드는 시간과 비용이 추가로 발생할 수 있다.

모델 검증
- 요구사항 분석 단계에서 만든 각종 모델이 요구사항을 제대로 표현하고 있는지 확인하는 활동
- 요구사항을 글로만 작성하면 사람마다 다르게 이해할 수 있기 때문에 검증하는 것이 좋다

모델 : 시스템을 이해하기 쉽게 표현한 그림 또는 구조

모델 종류
	ERD
	클래스 다이어그램
	시퀀스 다이어그램
	화면 흐름도
	API 명세서

핵심 목적
- 요구사항 반영의 확인 : 모델에 제대로 반영되었는지 확인
- 누락 발견 : 필요한 객체, 테이블, 관계 흐름이 빠졌는지 확인
- 오류 확인 : 잘못된 관계나 잘못된 흐름을 찾음
- 일관성 확인 : 여러 모델이 서로 맞는지 확인
- 구현가능성 확인 : 실제 코드와 DB로 만들 수 있는지 확인


4. 인수테스트
- 개발이 거의 끝난 후 만든 프로그램이 고객이 요구한 내용을 정말 만족하는지 최종 확인하는 테스트
테스트
- 코드의 동작 여부보다 고객의 요구사항 충족이 더 중요하다

과정
요구사항 작성 -> 인수테스트 계획 작성 -> 개발 진행 -> 시스템 완성 -> 인수테스트
-> 승인 또는 수정

종류
	사용자 인수 테스트
	운영 인수 테스트
	계약 인수 테스트
	규정 인수 테스트
	알파 테스트
	베타 테스트
