스키마 = 데이터베이스'
데이터를 어떻게 저장할 것인지에 대한 "설계도"

스키마의 구성요소
-테이블(table_
- 뷰(view)
- 인덱스(index)


데이터베이스는 좀 더 깊은 개념으로 3가지로 나눈다
1. 외부 스키마
- 사용자 이장에서 보는 구조
Ex) 화면에 보이는 데이터

2. 개념 스키마
- 전체 데이터 구조
- DB 설계자 관점

3. 내부 스키마
- 실제 저장 방식
- 디스크에 어떻게 저장되는지


DBMS의 종류
1. 계층형 데이터베이스
- 트리 구조로 데이터를 저장하는 방식
- 1:N 관계만 가능
- 구조가 단순하다
- 변경이 어렵다

2. 네트워크형 데이터베이스 
- 데이터를 그래프 형식으로 연결하는 방식
: N:N 관게가 가능하다
- 계층형볻는 유연하다
- 구조가 복잡하다
- 여러개의  연결을 하는  DB

3. 관계형 데이터베이스
- 테이블 형태로 데이터를 저장하는 방식
- 가장 많이 사용된다
- SQL을 사용
- 데이터 무결성을 보장한다

4. 객체지향 데이터베이스
- 객체 형태로 데이터를 저장
- Javam C++ 객체 그대로 저장 가능
- 복잡한 데이터 처리에 유리
- 사용 빈도 나즘

5. NoSQL(Not Only SQL)
- 관계형이 아닌 다양한 방식의 DB


DDL(Data Definition Language) : 데이터 정의어
- 데이터베이스 개체를 관리하는 언어
- 테이블, 속성, 제약조건, 뷰
- create, drop, alter, truncate




정규화
- ERD 내에서 중복 요소를 찾아 제거해나가는과정

1차 ~ 5차 정규화가 존재한다.

1정규형
- 한칸에는 하나의 값만 들어가야 한다

2 정규형
- 1정규형을 만족하면서 기본키의 일부에만 의존하는 특성을 분리한다

복합키
- 여러개의 속성을 묶어서 하나의 기본키로 사용하는 것
- 여러 컬럼의 값을 조합해서 겹치는 값이 없으면 된다

학생이름, 수강과목, 나이
	ㄴ학생이름, 나이
	ㄴ학생이름, 수강번호

3차 정규화
- 2차 정규화를 만족하는 상태에서 기본키가 아닌 다른 컬럼이 
	다른 일반 컬럼을 경정하면 분리한다

(PK)
학생번호, 학생명, 학과번호, 학과명, 학과사무실

학생 번호 -> 학과 번호
학과 번호 -> 학과명, 학과 사무실

학생 -> 학생 번호, 학생명, 학과번호
학과 -> 학과번호, 학과명, 학과사무실

이행함수 종속
A -> B, B -> C, A -> C가 성립되는 관계
중간 속성을 통해 간접적으로 결정되는 관계

정규화를 통해서 테이블을 너무 많이 쪼개게 되면 조회를 할 때 성능이 떨어진다
역정규화 하기도 한다
테이블을 만들기 전에 모델링을 잘 하는 것이 매우 중요하다
