## 현행 시스템의 분석

새 시스템을 만들거나, 기존 시스템을 개선하기 전에, 현재 사용 중인 업무 방식, 프로그램, 데이터, 서버,  
네트워크, 문제점을 조사하고 정리하는 활동


### 목적
  새로운 시스템을 정확하게 만들기 위해 현재 상황을 이해한다.  
  요구사항 확인 단계에서 현행 시스템 분석을 제대로 하지 않으면 다음 문제가 발생


| 잘못된 상황 | 발생 가능한 문제 |
| --- | --- |
| 기존 업무 흐름을 모름 | 필요한 기능을 빠뜨릴 수 있음 |
| 기존 데이터 구조를 모름 | 데이터 이전 시 오류 발생 |
| 기존 시스템 연동을 모름 | 외부 시스템과 연결 실패 |
| 사용자 불편 사항을 모름 | 새 시스템을 만들어도 만족도가 낮음 |
| 현재 문제 원인을 모름 | 문제 해결이 아니라 단순 재개발이 됨 |

### 현행 시스템 파악 절차

#### 1단계 : 현행 시스템의 구성, 기능, 인터페이스 현황 파악
무엇으로 구성되어 있고, 어떤 기능을 제공하며, 다른 시스템과 어떻게 연결되어 있는지 파악한다.  
1. 시스템 구성 현황 파악  
각 시스템이 어떤 업무를 담당하는지까지 정리해야한다.
2. 기능 현황 파악  
각 시스템이 제공하는 기능을 정리한다.  
3.  인터페이스 현황 파악  
시스템과 시스템이 데이터를 주고 받는 연결 지점을 의미한다.
```
  API(Application Programming Interface)
    - 프로그램끼리 서로 요청하고 응답하기 위한 약속
    - 시스템끼리 데이터를 주고 받기 위해 필요하다.
    - 외부 서비스와 연결하기 위해 필요하다.
```
#### 2단계 : 현행 시스템의 아키텍처 및 소프트웨어 구성 현황 파악
어떤 구조로 만들어져 있고, 어떤 소프트웨어를 사용하고 있는지 파악한다.
##### 왜 아키텍처 현황을 파악해야 하는가
1. 기존 시스템에 맞게 개발하기 위해서이다.
2. 변경 영향도를 파악하기 위해서이다.
3. 장애 원인을 찾기 위해서이다.
