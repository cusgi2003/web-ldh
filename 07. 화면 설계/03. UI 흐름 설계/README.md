# 1. UI 설계서 구성에 따른 작성 방법


## 1. UI 설계서 표지
- UI 설계서 표지는 문서의 첫 페이지이다.
- 어떤 프로젝트의 UI의 설계서인지 알 수 있도록 프로젝트명 또는 시스템명을 작성한다.
- 표지가 없으면 어떤 문서인지 구분하기 어렵다.


## 2. UI 설계서 개정 이력
- 개정 이력은 UI 설계서가 언제, 어떻게 수정되었는지 기록하는 부분이다.
- 처음 작성할 때는 보통 최초작성이라고 기록하고 버전은 1.0으로 설정한다.
- 버전은 문서의 수정 상태를 나타내는 번호이다.


## 3. UI 요구사항 정의
- 사용자가 원하는 화면 기능과 조건을 정리하는 부분이다.

기능 요구사항과 비기능 요구사항

| 구분 | 설명 | 예시 |
| --- | --- | --- |
| 기능 요구사항 | 사용자가 수행할 수 있어야 하는 기능 | 로그인, 검색, 저장 |
| 비기능 요구사항 | 기능의 품질이나 조건 | 버튼 크기, 색상, 응답 속도 |


## 4. 시스템 구조
- 서비스 전체가 어떤 메뉴와 기능으로 구성되는지 나타낸 것이다.
- UI 요구사항과 UI 프로토타입을 참고하여 UI 메뉴 구조를 설계한다.
- 전체 화면의 범위를 한 눈에 알아볼 수 있다.
- 중복되거나 빠진 메뉴를 찾기 쉽다.
```
  1층 : 로그인
  2층 : 메인
  3층 : 출결관리, 상담관리, 통계 관리
```


## 5. 사이트 맵(Site Map)
- 시스템 안에 포함되는 메뉴와 화면을 구조적으로 정리한 그림 또는 표이다.
- UI구조에 포함되는 메뉴를 사이트 맵 형태로 작성하기, 상세 내용은 도표 형태로 작성한다.
```
  메인
  ├ 공지사항
  │  ├ 공지사항 목록
  │  └ 공지사항 상세
  ├ 게시판
  │  ├ 게시글 목록
  │  ├ 게시글 작성
  │  └ 게시글 상세
  └ 마이페이지
     ├ 내 정보
     └ 비밀번호 변경
```


## 6. 프로세스(Process) 정의
  - 사용자가 서비스를 이용할 때 어떤 순서로 행동하고, 시스템이 그 행동에 어떻게 반응하는지 정리하는 작업이다.
  - 사용자가 목적을 달성할 수 있도록 순서를 설계해야 한다.
  - 사용자 행동 순서와 시스템 처리 순서를 명확히 만드는 작업이다.
```
  프로세스 정의표
      - 업무나 기능이 어떤 순서로 수행되는지 표 형태로 정리한 문서
      - UI 흐름 설계에서는 사용자의 행동과 시스테의 처리 과정을 단계별로 정리하기 위해 사용한다.
```


## 7. 화면 설계
- 프로토타입과 프로세스 정의를 참고하여 실제 필요한 화면을 구체적으로 설계하는 단계이다.
- 각 화면별로 구분되도록 고유 ID를 부여하고, 별도의 표지 페이지를작성한 후 화면별 내용을 설계한다.  
화면 설계의 목적
1. 개발자와 디자이너의 공통 기준 제공
2. 누락된 기능 발견
3. 사용자 경험 검증

화면 ID를 부여하는 이유
- 화면이 많아지면 이름만으로구분하기 어렵기 때문이다.



# 2. UI 화면 설계의 기본 구성요소
```
  아이디 입력창
  비밀번호 입력창
  로그인 버튼
  회원가입 링크
```
- 이러한 요소를 UI 구성 요소(UI Component)라고 부른다.
- 화면 설계를 하기 위해서는 어떤 UI 구성 요소가있는지 알아야 한다.

ISO 9241-161
- 사용자와 컴퓨터의 상호작용에 대한 국제 표준
- 화면 설계 시 자주 사용되는 기본 UI 구성 요소를 정의하고 있다.
- UI 기본 구성 요소를 버튼, 체크박스, 입력창 등 51개 항목으로 제시한다.
```
컨테이너, 버튼, 체크 박스, 라디오 버튼, 입력창, 레이블, 목록, 대화상자, 폼
```
> 체크 박스 vs 라디오 버튼  
> 체크 박스는 중복 허용, 라디오 버튼은 단일 선택
